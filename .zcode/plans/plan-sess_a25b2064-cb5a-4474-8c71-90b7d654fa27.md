# 项目工具台加速：快找 + 并行测

## 目标
让 Agent 用 MCP **少搜、快改、并行验**，砍掉「一个个串行等 pytest / 瞎找代码」的时间。

不在本轮：升级 pip `mcp` 到 2.x、恢复历史反馈 md、大改 release_gate。

## 现状（已摸清）
- `run_test`：单次 **一个 pytest 进程**；`target` 可空格多路径但仍串行。
- `tool_job_submit`：多 job 可并发，但 Agent 常只串行 `run_test`。
- `lint`：多路径 **串行**；目录会报「文件不存在」。
- 找代码：`code_explore/node/impact` 薄封装；**无 rg**；`codegraph affected`（改动→测试）未暴露。
- 已有 `context_load`（并发装弹），SOP 仍偏串行证据工具。

## 方案：三个可复用能力 + 文档/SOP

### 1. 并行测试组（核心）
**新工具 `run_tests_parallel`**（放在 `dev_toolkit/code_tools.py`，与现有 `run_test` 同模块）

| 参数 | 说明 |
|------|------|
| `targets` | 字符串（空格/逗号/JSON list）或多个目标 |
| `max_workers` | 默认 `4`，上限 `8` |
| `timeout_per_target` | 默认 `120` |
| `mode` | `auto`（默认）/ `force_parallel` / `serial` |

**行为**
1. 复用 `normalize_pytest_targets` / `pytest_targets_for_command` / 现有 pytest venv。
2. 拆成 **每个 target 一次 pytest 进程**（进程级隔离，比 xdist 更稳）。
3. `asyncio.Semaphore(max_workers)` + `gather` 并行跑。
4. **`mode=auto` 分桶**（避免共享库/live 互打）：
   - **串行桶**：路径含 `e2e` / `live` / `release_gate`，或节点标记可探测时的 `serial_db`（尽量启发式，不依赖未使用 marker）。
   - **并行桶**：其余（尤其 `dev_toolkit/test_*.py`、纯 unit）。
   - 串行桶在并行桶之后顺序跑，或单独串行队列，避免抢库。
5. 返回紧凑汇总 JSON：
   - `success` / `passed` / `failed` / `duration_seconds`（墙钟）
   - `results[]`：每目标 `success/returncode/duration/stdout_tail/stderr_tail/command`
   - `parallel_count` / `serial_count` / `max_workers`

**兼容**
- 保留 `run_test`（单目标 / 兼容 finish_task）。
- `tool_job_tools.SUPPORTED_TOOLS` 增加 `run_tests_parallel`，长跑可后台提交。
- 更新 `run_test` description：写明可多 target，但并行请用新工具。

### 2. 改动 → 该测什么（减少瞎选 target）
**新工具 `select_verify`**（可同文件或小模块 `verify_tools.py`）

输入（任选）：
- `paths`：改过的文件列表；或
- `from_git=true`：用现有 `worktree_tools.git_changed_entries` 取 dirty；
- `extra_targets`：人工追加。

输出：
```json
{
  "lint_paths": [...],
  "test_targets": [...],
  "parallel_safe": [...],
  "serial_preferred": [...],
  "notes": ["heuristic | codegraph_affected | missing_test_guess"]
}
```

映射启发式（先稳后全）：
- `dev_toolkit/foo.py` → `dev_toolkit/test_foo.py`（存在则收）
- `backend/app/.../x.py` → `backend/tests/**/test_*x*` 或同名 test（glob 有限）
- `modules/{key}/...` → `modules/{key}/**/test_*.py` 与 `sandbox/`（有上限）
- 若本机有 `codegraph affected`：优先用其结果，失败则纯启发式

Agent 标准闭环：`select_verify` → `lint(paths)` → `run_tests_parallel(targets)`。

### 3. 快找：`rg_search` + 加强 codegraph 入参
**新工具 `rg_search`**（`dev_toolkit/search_tools.py` 新组件）

- `pattern`, `path`（默认仓库根）, `glob`（如 `*.py`）, `max_matches`（默认 30）, `context`（0–3）
- 固定排除：`node_modules`, `.venv`, `backend/.venv`, `dist`, `.git`, `memory_embeddings.json`, 大缓存
- 输出结构化：`file:line:text` + 命中计数；路径限制在仓库内（复用 `resolve_repo_path` 思路）
- 实现：优先 `rg` CLI，无则 `grep -R` 降级

**增强现有**（小改 `code_tools`，不拆三工具）
- `code_explore` 增加可选 `path` / `max_files`（转发给 codegraph 若 CLI 支持，否则忽略并在结果里注明）
- `code_node` 增加可选 `file`（有则带 `--file`）

### 4. lint 小修（顺手，反馈高频）
- `lint_one`：允许 **目录** → `ruff check <dir>`（解决「目录不存在文件」）
- 多路径改为 **有限并行**（`max_workers=4`），与并行测一致

### 5. SOP / 版本
- `dev_toolkit/README.md`：验证表改为
  `context_load` → `rg_search` / `code_*` → edit → **`select_verify` → `run_tests_parallel`**
- `SERVER_VERSION`：`1.0.0` → **`1.1.0`**
- `plan_task` 证据清单（`server.py` `_build_evidence_checklist`）：首推 `context_load` + `rg_search`，验证步写 `select_verify` + `run_tests_parallel`

### 6. 接线与测试
- `server.py`：`list_tools` / `call_tool` 注册 `search_tools`；`code_tools` 新工具随现有 code 路由自动挂上
- 单测（假 `run_command_json` / 临时目录）：
  - `test_run_tests_parallel_*`：2–3 个 fake target 并行、失败汇总、max_workers
  - `test_select_verify_*`：git/path 启发式
  - `test_rg_search_*`：仓库内命中 + 排除
  - lint 目录路径
- 本地：`pytest dev_toolkit/test_*.py` 用 **新工具自测**（并行跑相关文件）

## 刻意不做
- 默认全局 `pytest -n auto`（共享 DB/live 易 flaky；xdist 可作后续可选参数）
- MCP 2.x / 无界 job 线程池重构
- 自动跑 full `release_gate`

## 实施顺序
1. `run_tests_parallel` + 单测
2. `select_verify` + 单测
3. `rg_search` + codegraph 可选参数
4. lint 目录/并行
5. server 接线、README、`SERVER_VERSION`
6. 用并行工具跑一遍相关 `dev_toolkit` 测试验收

## 成功标准
- 一次 MCP 调用可 **并行跑多组 pytest**，墙钟明显短于串行相加
- 改几个文件后，`select_verify` 能给出可直接喂给并行测的 target 列表
- `rg_search` 可在限定路径内秒级定位字符串，无需 Agent 反复 shell 试探
- 现有 `run_test` / finish_task 行为不破坏
