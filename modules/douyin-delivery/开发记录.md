# 抖音投放助手 (douyin-delivery) 开发记录

## 做了什么

- 创建了 `modules/douyin-delivery/` 完整模块（manifest + 前端 + 后端 + sandbox）
- **后端（Python FastAPI）**：5张业务表（`douyin_products/scripts/ad_copies/campaigns/prompts`），Router → Service → Model 三层架构，17个 API 端点
- **前端（Vue3 + Element Plus）**：5个 Tab 页（口播脚本、广告文案、投放计划、产品管理、内容校验、提示词设置），表格 CRUD + AI 生成
- **跨模块能力注册**：`generate_script`、`generate_ad_copy`、`validate_content` 三个能力注册到框架能力注册表
- **底座接入**：模型网关（`gateway.service.chat`）做生成 + 知识库（`call_capability("knowledge","search")`）做内容校验
- **提示词 DB 化**：7个默认提示词（人设/脚本/文案/渠道说明/校验）种子到 `douyin_prompts` 表

## 关键决策

1. **模块 key = `douyin-delivery`**（语义清晰，覆盖全部投放场景）
2. **三合一 MVP**：口播脚本 + 广告文案 + 投放计划一次覆盖，华哥说三个都需要
3. **渠道颗粒度从一开始就做细**：`local_push`/`ocean_engine`/`qianchuan` 三个渠道各自有提示词/API 参数
4. **数据表设计**：5表全以 `douyin_` 前缀，`owner_id` 隔离，无外键约束到框架表
5. **提示词存 DB + fallback**：从 DB 动态加载，DB 没有就走 `DEFAULT_FALLBACKS` 兜底，不改代码即可调优

## 踩的坑

1. **模块注册需重启后端**：manifest 新模块不会热加载，必须 `--restart` 重启后端。已在 AGENTS.md 和脚本中确认流程
2. **认证**：API 端点默认需登录鉴权，测试时忘了加 `Authorization` header
3. **sandbox vite 端口冲突**：默认 5173 已被框架占用，改为 `VITE_SANDBOX_PORT=5175` 等
