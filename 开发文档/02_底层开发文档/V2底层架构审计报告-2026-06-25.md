## 1. 全局结论（最该先清的 3 个地基问题）

1. **5 个 parser 模块是模板化拷贝，80% 重复代码挡着统一 Document IR 的路（A11 前置）**  
   - 每个 parser 都有逐字拷贝的 `_resolve_user_id()`、文件路径校验、`ParseRequest`、health endpoint、`register_capability(…, "parse", …)` 四件套  
   - 返回的 JSON schema 各不相同——docx 有 `resources` 顶层字段，text 没有；docx 的 block 有 `page`，xlsx 没有——A11（统一文档 IR）之前必须统一  
   - 同时框架层 `backend/app/services/office/` 还保有一份 `DocxService.parse()` / `PptxService.parse()`，和模块层 parser 做同一件事  

2. **`module_registry` 和 `module_events` 是两张互相不感知的能力/事件注册表**  
   - 当前 `emit_module_event()` 只有 1 个调用者（`file_transfer.py:upload`），实质是死代码  
   - 大部分跨模块通信走 `call_capability`，但也有些模块直接在 handler 里硬编码 `knowledge:search` 字符串（虽不算违规，但说明 registry 的元数据没被前端消费）  
   - A9 统一控制面之前，必须决定它们是合还是拆  

3. **模型调用有 4 层嵌套的 retry/fallback 叠加，但各层彼此不感知对方的重试状态**  
   - 调用链：`_call_with_retry`（同 provider ×3 重试）→ `gateway.chat()` → `chat_with_fallback`（跨 profile 降级链）→ `chat_with_degradation_chain`（顶层 catch-all）  
   - 这四层分别在不同文件（`gateway/router.py`、`fallback_chain.py`、`engine.py`）里各自独立实现  
   - 结果：一次模型调用最多可以跑 `3 retries × N profiles × 1 catch-all = 3N+1` 次，但没有任何一层知道其他层的尝试历史

## 8. 不确定项（请你决定）

1. **`signals.py` 的去留**  
   `SignalBus` 当前只消费了 `memory_recall_quality`（`engine.py:98`）。其他 4 个信号类型从未被 emit。它是 A3（记忆精修）的前置基础设施，还是过早抽象？  
   → **如果你确认 A3 会用全这 5 个信号**，保留。否则我建议先删掉未使用的类型，等 A3 再建。

2. **知识库 entity_service.py 的两条抽取路径**  
   `process_document_entities` (line 106) 从原始 blocks 抽实体，`process_document_entities_from_fusions` (line 304) 从融合层抽实体。两者 60% 逻辑重复。  
   → **fusion 版会完全替代原始版吗？** 如果是，删原始版并统一接口；如果两者长期共存，抽共享函数 `_extract_and_save_entities()`。

3. **Capability 注册表 vs Manifest 声明的关系**  
   `module_registry.py` 的运行时元数据（`min_role`/`parameters`/`description`）和 `App` 表的 `capabilities`/`public_actions` JSON 字段是 **两份数据**。  
   → **manifest 声明应该作为唯一事实源**（运行时注册时自动校验），还是保持现状（各管各的）？决策影响 A9 control plane 的设计范围（如果是前者，A9 就多一个"manifest → registry 同步"的任务）。

4. **watchdog 的 model registry 能否和 gateway 的 model config 合并**  
   `watchdog/registry.py` 从 `models.json` 的 `watchdog_models` 段加载；`gateway/config.py` + `gateway/service.py` 从 `models.json` 的 `llm`/`vision`/`embedding` 段加载。  
   → **这是故意的分层还是该合并？** 如果是故意的（watchdog 只管本地模型启停，gateway 只管 API 调用），当前结构可以保留但至少应共用 `models.json` 文件路径和解析函数。

5. **`engine.py` 的 `assemble_context` 职责边界**  
   当前它既做编排（调用子模块），也包含具体检索逻辑（`_find_skills`、`_three_layer_recall`、`_experience_match` 的直接调用）。  
   → **A1 是否要把 assemble_context 的"检索"执行权全部委托给各个子模块的服务类**（如 `skills_loader.find_skills()` → 返回匹配列表；`layered_memory.three_layer_recall()` → 返回注入文本）？当前已经部分这么做了，但 engine.py 还直接持有具体函数的引用。A1 是否要求彻底解耦？
