## 一、这批信是什么

这是把 V2 Agent 系统的整套升级方案，拆成一封一封**可独立执行**的任务信。
每封信只解决一个节点，不许顺手扩题。

## 二、这批信的真实背景（先看这个，别急着写代码）

审计已经确认：V2 现在不是“没能力”，而是**底座有重复实现，地基不平**。
如果直接往上加能力，会把脏东西一起搬过去。所以这批信**前面几封是清地基，后面才是加能力**。

审计点名的三大地基问题：

1. **双套后台可靠性机制并存**
   - `modules/agent/backend/engine/post_turn_hooks.py:504-707` 有一套 leader 选举（维护循环单主执行）
   - `backend/app/services/task_worker.py:57-230` 有另一套抢占回收（`FOR UPDATE SKIP LOCKED`）
   - 两套解决的是同一类问题，字段、回收逻辑都不统一 → **信0 先合并它**

2. **chat.py 运行时编排职责过载**
   - `modules/agent/backend/handlers/chat.py` 的 `event_stream()`（约 146-571，单函数 400+ 行）一个函数干 8 件事
   - → **信1（A1）拆它**

3. **模型治理四处分散 + fallback 双实现**
   - `gateway/config.py` + `gateway/service.py` + `gateway/router.py` + `agent/engine/fallback_chain.py`
   - → **信5（A5）收口它**

## 五、统一调研纪律（每封信都适用，不再重复写）

V2 自带项目工具台（`dev_toolkit/`），开工前**必须先用**，不许凭感觉改代码：

1. `brief` —— 先拿项目全景，别空降
2. `codegraph explore/node/impact` —— 查符号、调用链、改动影响面。**改任何函数前先查 impact**
3. `routes` / `capabilities` / `db_schema` —— 查准接口、能力名、表结构，不要猜
4. 改完用 `probe` / `call_capability` 打活系统，再 `run_test` 跑针对性测试
5. 收工用 `memory_write` 写回项目记忆 + 错误归因

**有 `.codegraph/` 索引就用它，禁止 grep 逐文件翻。**

## 六、统一红线（每封信都适用）

- **模块隔离**：只动本信授权的文件/目录。跨模块只能走对外接口（capability registry），禁止直接 `require/import` 别人内部子文件。
- **不许扩题**：本信只解决一个节点。看到顺手能做的其他节点，记下来，但不做。
- **不破坏现有契约**：`/api/agent/chat` SSE、`agent_events`、各 capability 名、现有测试，不许退化。
- **每条改动带证据**：说“这里要改”必须给 `文件:行号`。没用 codegraph 查过影响面的结论，标注“未验证影响面”。
- **完整交付**：禁止只给骨架、占位符、TODO。同事拿到直接能跑。

## 八、华哥的验证协议（给上游看的，不是给执行者的）

活交给执行者，但每封信回来后我（上游）要验证，不能全信“做完了”：
1. 回信里的 `文件:行号` 抽样核对，看改动是否真落在该落的地方
2. 验收清单逐条让执行者给出**活系统证据**，不接受“应该可以”
