# 任务信：信10 — A10 项目工具台 / CodeGraph / Codemap 吸入默认工作流

> 第四层（平台化收口）第 4 封。前置：信1（runtime policy 注入点）、信2（技能注入框架）。
> 依据：总图节点 A10（2373-2430）+ 草稿信 6。

---

## 任务目标

本信只解决一个节点：把 `brief → codegraph → probe → memory_write` 固化成 Agent 默认工作流。
**不**先扩很多新工具，不做新前端壳。

## 为什么现在做这个

- V2 已有很强的项目工具台，但它还是“会用的人主动去用”，不是 agent 默认心智。
- 真正厉害的不是“工具存在”，而是 agent 什么时候用、怎么选、用完怎么沉淀。
- 如果默认工作流不固化进 agent 侧，后续再强的技能/知识库/记忆都会被不稳定的工作方法抵消。

## 当前事实（V2）

- 文件：
  - `dev_toolkit/server.py:611-650` codegraph 包装层
  - `dev_toolkit/server.py:823-1117` `brief` / `probe` / `call_capability` / `run_test`
  - `dev_toolkit/server.py:1117` `memory_write`
  - `dev_toolkit/smoke.py:89-467` 红绿矩阵 smoke
  - `modules/agent/backend/engine/engine.py:234-283` `_build_system_content()`（默认 workflow 可写进去）
  - `modules/agent/backend/engine/skills_loader.py:20`
- 当前已具备：code_explore/node/impact、routes、capabilities、db_schema、probe、call_capability、run_test、memory_write、smoke
- 当前缺口：
  - agent 没把“先 brief、再 codegraph、再 probe、最后写 memory”当默认执行骨架
  - 工具台外部能力强，但没变成 agent 内建方法论
  - 缺“有没有按正确工作流调研/验证”的自检和可观测反馈

## 参考事实（外部更强实现）

- Claude：工程型工作流稳定，调研/执行/验证像内建行为
- Hermes：持续学习自改进（适合作 review/evolve 思路）

## 本次要改什么

### 1. 默认工作流收口成明确阶段
```
brief → codegraph → runtime facts(routes/capabilities/db_schema)
     → implementation → probe/call_capability → targeted test/smoke → memory_write
```

### 2. 注入默认工作方法
- `engine.py:234-283`：让技能系统 / system content / runtime policy 能显式注入这条工作方法
- 不再靠每次提示词提醒

### 3. 增加最小自检信号
本轮是否：
- `did_brief`
- `did_codegraph`
- `did_runtime_verify`
- `did_live_probe`
- `did_write_memory`
- `maturity_level`

### 4. 错误归因 + 命中反馈回写
- 错误归因记录
- 命中率反馈记忆
- CodeGraph/Codemap 不准时的回写协议

## 本次不要改什么

- 不新增一整套平行工具系统
- 不把 codemap/codegraph 再做成重复产品
- 不强制所有简单问答都走重型流程
- 不破坏现有 dev_toolkit 独立可用性

## 精度控制手柄

- 先查 codegraph，再查 codemap，再逐文件读
- 先 probe/call_capability，再写测试脚本
- 事实不足时强制进入 evidence-gathering 模式
- 简单任务不被无意义重流程拖慢

## 验收

### 单测
- `dev_toolkit/smoke.py`（不退化）
- 新增 agent runtime 工作流注入测试

### 活系统
- 项目型任务默认能看到 CodeGraph / probe / memory_write 被优先采用
- 简单任务不被重流程拖慢（实测）
- 执行后能判断这轮工作到了哪个成熟度等级

### 行为口径（做完必须全部成立）
1. Agent 侧存在清晰的项目默认工作流注入点
2. 项目型任务默认先查 impact 再改代码
3. 事实不足时会先 probe / call_capability
4. 事后能写回项目记忆与错误归因
5. 简单任务不会被无意义重流程拖慢
