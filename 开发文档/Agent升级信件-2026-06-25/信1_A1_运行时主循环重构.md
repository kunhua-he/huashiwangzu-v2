## 任务目标

本信只解决一个节点：把 `chat.py` 里的运行时编排外提，拆成 4 层 runtime。
不要顺手做理解环（信6）、技能治理（信2）、模型治理（信5）、知识库升级（信4）。

## 为什么现在做这个

- V2 的真实问题不是不能跑，而是 `chat.py` 的 `event_stream()` **一个函数 400+ 行，同时干 8 件事**：
  1. SSE 适配（yield 格式化）
  2. 工具循环主调度（MAX_TOOL_ROUNDS）
  3. inline tool recovery（内联工具调用恢复）
  4. stuck detector / diminishing returns 止损
  5. assistant 消息持久化
  6. timeline 持久化
  7. post-turn hooks 触发
  8. 事件落库
- 审计确认其严重程度**超出总图预期**（单函数 425 行，职责 8 项）。
- 如果不先拆 runtime，后续任何理解环 / review fork / 角色模型接入都会继续往 `handle_chat()` 里堆，越堆越死。

## 当前事实（V2）

- 文件：
  - `modules/agent/backend/handlers/chat.py:39-99` `_yield_final_stream()`：既负责 token 流，也负责 inline tool recovery
  - `modules/agent/backend/handlers/chat.py:101-571` `handle_chat()` + `event_stream()`：上面 8 项职责全在这里
  - `modules/agent/backend/handlers/chat.py:168-171` `MAX_TOOL_ROUNDS` 工具回合控制
  - `modules/agent/backend/handlers/chat.py:430-478` stuck detector / diminishing returns 打断
  - `modules/agent/backend/engine/engine.py:32-188` `assemble_context()`：上下文装配，但也承接了部分运行时编排
  - `modules/agent/backend/engine/engine.py:307-317` `get_hooks()`：已经是 runtime 生命周期挂点
- 当前缺口：
  - 没有显式 runtime service
  - 没有可复用的 stream emitter / tool loop / task sink
  - review fork、understanding loop、task 化角色都没有清晰挂点

## 本次要改什么

### 1. 新增 runtime 目录
新建 `modules/agent/backend/runtime/`，最少拆出 4 个文件：

```
modules/agent/backend/runtime/
  conversation_runtime.py   — 会话生命周期编排（取代 event_stream 的总调度职责）
  tool_loop_runtime.py      — 工具循环 + 止损（MAX_TOOL_ROUNDS / stuck / diminishing）
  stream_emitter.py         — SSE 流式输出 + inline tool recovery（取代 _yield_final_stream）
  task_sink.py              — 事件/消息/timeline 持久化（依赖信0 的 DurableLease）
```

### 2. chat.py 瘦身
- `handlers/chat.py` 缩到只负责：认证 → payload 校验 → 调 runtime → 返回 `StreamingResponse`
- 不再在 handler 里写工具循环、流式恢复、持久化逻辑

### 3. engine.py 归位
- `assemble_context()` 保留上下文装配与 hook getter 职责
- 不再承接运行时编排，编排移到 `conversation_runtime.py`

### 4. 不改外部契约
- `/api/agent/chat` SSE 行为不变
- `agent_events`、`agent_message_meta.timeline` 语义不变

## 本次不要改什么

- 不要顺手做 understanding loop（信6）
- 不要改 knowledge 检索能力（信4）
- 不要改 gateway 模型治理（信5）
- 不要改 prompt 资源系统
- 不要新增复杂策略——本次只要求把已有手柄“收口”，不新增

- `MAX_TOOL_ROUNDS`（现 `chat.py:168-171`）

- diminishing returns（现 `chat.py:469-478`）
