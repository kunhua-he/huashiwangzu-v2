## 任务目标

本信只解决一个节点：把技能从“只读扫描器”升级成“可治理对象”。
不要一口气把技能 hub / 市场都做了。本信只做 CRUD + approval + usage 这一层。

## 为什么现在做这个

- 技能、记忆、知识库是主模型的三类认知材料。本封先治理技能。
- 当前技能只能读（扫盘 → 注入），不能写、不能审批、不能沉淀使用情况。
- 用户纠错、review fork 产出的好做法，现在无法沉淀成可复用技能。
- 如果不先把技能治理清楚，后续多代理和模型治理会放大噪音，而不是提高命中率。

## 本次要改什么

### 1. 新增 capability：`agent:skill_manage`
- `create` / `update` / `delete` / `list_versions` / `rollback`

### 2. 扩 skill metadata 字段
在 `skills_loader.py` 的 `SkillDef` 和落库模型上新增：
- `source`（来源：内置 / 用户 / review 产出）
- `approved_by`
- `scan_status`（安全扫描状态）
- `usage_count`
- `last_used_at`
- `model_override`
- `hooks`

### 3. 注入逻辑改造
- `engine.py:100-116`：从“扫盘后直接注入”改成“先查 registry，再按策略注入”
- 未审批技能**禁止自动注入**

### 4. 新增 skill 扫描与审批表
- 存技能正文、版本、审批状态、扫描结果

### 5. review fork 只产 proposal
- review fork（信6/post_turn_hooks 体系）产出的技能只能进 proposal 状态，**不能直接写正式 skill**

## 本次不要改什么

- 不要做技能市场 / hub UI
- 不要改 runtime 主循环（信1 已定型）
- 不要破坏现有 `<available_skills>` prompt 形态（注入格式兼容）

## 精度控制手柄

技能不是越多越强，要控：
- 注入数量上限
- 按任务类型过滤
- 低质量技能（usage 低 / 久未用）降权
- 未审批技能禁止自动注入
- 复用 `engine.py:98-116` 现有的信号感知注入框架
