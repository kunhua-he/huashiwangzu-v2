# 任务信：信3 — A3 记忆精修

## 任务目标

本信只解决一个节点：把记忆从“召回多”升级成“选得准”。
先做 selector / fencing / frozen snapshot，**不**先做 memory provider 大重构。

## 为什么现在做这个

- 记忆、技能、知识库是三类认知材料，本封治理记忆。
- 当前短板不在“没有记忆”，而在“召回后如何选给主模型看”——召回强，surfacing 不够精细。
- 系统真正提精度靠“选对记忆”，不是“塞更多记忆”。长会话里记忆越来越吵。

## 本次要改什么

### 1. 加 memory selector
- `layered_memory.py:436-571` recall 阶段增加二次排序与裁剪
- 召回 → selector（relevance/weighting/pruning）→ 才注入 system prompt

### 2. 加 memory fencing
- `layered_memory.py:571-675` `three_layer_recall()` 和语义注入段接 fencing / pruning
- 新增 `memory_fence_reason`（为什么这条没进上下文）

### 3. 加 frozen snapshot
- `models.py:168-176` `ContextSnapshot` 扩成 frozen session snapshot 持久化基础
- 新增 `session_snapshot_type = frozen_context`，长任务冻结记忆，减少中途漂移

### 4. surfacing budget
- `engine.py:137-153` 三层注入点加 selector 输出
- `engine.py:170-179` experience 注入与 recall 统一做 surfacing budget

### 5. review fork 只产 proposal
- review fork 只生成 memory proposal，**不直接污染主记忆**

### 6. 顺手清理 signals.py（审计 §7 P2 行179）
- `modules/agent/backend/engine/signals.py:44-50` 当前定义了 `degradation_status`、`tool_failure_rate`、`diminishing_returns` 三个**从未被 emit** 的信号类型
- 本信若用到记忆命中反馈信号，就补全实装；用不到的，删掉未使用类型（不要留虚假信号）

## 本次不要改什么

- 不要做 memory provider 大重构
- 不要改 runtime 主循环（信1 已定型）
- 不要改 knowledge 检索（信4 的活）

1. 平均注入字符数下降，但有效命中率上升
2. recall quality 指标更稳定
3. 长会话中“记忆越来越吵”现象下降
4. review fork 只操作 proposal 层，不污染主记忆
5. signals.py 不再有从未 emit 的虚假信号类型
