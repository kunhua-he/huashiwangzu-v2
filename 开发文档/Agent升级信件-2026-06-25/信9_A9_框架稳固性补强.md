## 任务目标

本信只解决一个节点：把散落在多处的稳定性补丁，收口成统一的 health / trace / checkpoint 模型。
先补统一诊断和健康 contract，**不**先做前端大盘。

## 为什么现在做这个

- 信0 已经把两套 lease 合并了。但 V2 的稳固性还散落在多个点：task_worker、gateway、module_registry、module_events、post_turn_hooks 各有一套可用机制，没形成统一 durability/health/observability 语言。
- 如果不补这一层，后续 A1/A4/A5/A6 做得越重，越容易出现“功能更强，但定位更难、回收更难、重复执行更难”。
- 这一层做完，后续所有 agent 升级都复用这个底座，不再各自补稳定性逻辑。

### 2. 统一 trace / correlation id

### 3. 后台任务和 hook 补诊断字段
- `lease_owner`（来自信0）
- `retry_reason`
- `state_version`
- `checkpoint_ref`

### 4. 副作用 / 幂等声明
为 task / event / capability / maintenance loop / gateway route 补：
- `side_effect_level`
- `idempotency_level`
- `failure_scope`
- `takeover_allowed`

### 5. 统一 admin 视图
至少新增一个 admin 视图或服务输出，能同时看见：
- 当前 worker/leader
- 最近心跳
- 最近失败（含原因）
- 是否可接管
- 最近重复执行/重试情况

## 本次不要改什么

- 不要重构整个队列系统成外部中间件
- 不要引入 Redis / Kafka / Celery 级别大迁移
- 不要改业务模块自己的业务逻辑
- 不要把 observability 做成大型前端平台

## 精度控制手柄

框架层的“提精度”是执行可靠性，不是 prompt：
- 某任务是否允许自动重试
- 某 hook 是否允许 fire-and-forget
- 是否必须持久化成功后再推进状态
- 某能力失败时是否允许降级
- 某文件读取是否必须经过 evidence boundary
