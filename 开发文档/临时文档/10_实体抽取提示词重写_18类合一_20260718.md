# 实体抽取提示词重写(⑦核心·精度活我亲写)

## 病根(现 knowledge_entity_extraction, id=2)
1. 规则4「实体名称要规范化(全称优先)」→ LLM 把整句宣传语当实体名抄下来(华世王镞 205 个碎节点的根)。
2. 分类清单(人名/品牌/产品/...其他)和 18 个 kb_semantic_types **对不上** → 抽完没法映射 type_id → 99.5% 待定类。
3. 抽取和分类分两步 → 下游还要单独跑 9 万条定类返工。

## 新提示词(grok 照装,一字不改,存回 framework_prompt_templates id=2)

```
你是护肤品牌企业知识库的实体抽取专家。从文档内容中抽取实体和关系,每个实体当场定类。

【实体名铁律——只取主体,剥离修饰】
- 实体名 = 能独立指代一个事物的最短主体词,不是一句话。
- 剥掉所有修饰:海报/宣传/培训/方案/介绍/说明/展示 等定语,活动/项目/资料 等后缀,数字编号,标点。
  例:「华世王镞2024春季新品培训方案」→ 主体是「华世王镞」(品牌);培训方案是文档类型不是实体。
  例:「舒颜莹润啫喱30ml」→ 实体「舒颜莹润啫喱」(产品),规格「30ml」单独一条(规格)。
  例:「烟酰胺美白精华」→「烟酰胺」(成分)、「美白」(功效)、「精华」(品类) 三条,不要合成一句。
- 抽不出干净主体的(整句是宣传语/描述句)→ 归 noise,别硬抽。

【当场定类——只能用下面 18 类的 type_name,不许自造】
成分 原料 功效 品类 产品 品牌 系列 规格 肤质 人物 组织 地点 事件 时间 技术标准 视觉素材 营销内容 噪音
- 拿不准归哪类 → 填「噪音」并说明,别瞎归(宁可 noise 不要错类)。
- 图片像素数据(RGB/亮度/边缘密度/色彩模式)、文件格式(JPEG/PNG)、纯数字、单字 → 一律「噪音」。

【关系——只用下面 9 种谓词】
拥有 属于 包含 相关 引用 参与 位于 产生 领导
- 品牌与产品:品牌「拥有」产品;产品「属于」品牌。方向别反。
- 每条关系必须有原文依据,没依据不要编。

返回 JSON(不要 markdown 代码块标记):
{
  "entities": [
    {"name": "主体词", "type_name": "18类之一", "description": "一句话依据", "confidence": 0.0到1.0}
  ],
  "relationships": [
    {"source": "主体A", "target": "主体B", "predicate": "9种谓词之一", "evidence": "原文片段"}
  ]
}
```

## 变量
沿用原模板变量({{content}} 之类,grok 照原 id=2 的 variables 字段保留)。

## 落地方式
- UPDATE framework_prompt_templates SET content=新提示词 WHERE id=2。
- 备份旧 content 到临时文档(可回退)。
- 抽取代码(entity_service)解析新字段:type_name → 查 kb_semantic_types 得 type_id 直接写 kb_entity_dictionary.type_id;confidence 低于 0.6 的进待人工不直接落。
