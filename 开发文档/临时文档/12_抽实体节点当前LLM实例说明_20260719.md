# 抽实体节点：当前到底怎么打 LLM（流程 + 真实实例）

> 只讲 **节点⑦ 抽实体**，不讲融合页。  
> 目标：看懂“每次发给模型的是什么、模型回了什么”，方便后面改算法。  
> 实例来自 2026-07-19 对文档 `4269` 第 1 页的真实调用（模型：`grok-4.5` @ `127.0.0.1:8317`）。

---

## 1. 先记住一句话

**当前抽实体 = 按页打。**

- 一篇文档有多少条「融合页正文」
- 就发多少次 LLM
- **不是**整篇文档只打 1 次
- **不是**一句话一句话打

---

## 2. 文本流程图（只看抽实体）

```text
开始处理某篇文档（graph 阶段）
        │
        ▼
从数据库取出该文档所有「融合页正文」
（表：kb_page_fusions，条件：正文不为空）
        │
        ▼
按页循环（第1页、第2页、第3页……）
        │
        ├──────────────┐
        ▼              │
拼一次 LLM 请求：      │
  系统提示词（固定）  │
  + 用户正文（这一页） │
        │              │
        ▼              │
发给大模型             │
        │              │
        ▼              │
模型返回：实体 + 关系  │
        │              │
        ▼              │
解析结果，先攒着       │
        │              │
        └──── 下一页 ──┘
        │
        ▼
这一文档所有页都抽完
        │
        ▼
统一落库 / 归并 / 建图
        │
        ▼
结束
```

### 关键点

| 问题 | 当前答案 |
|---|---|
| 一次 LLM 处理范围 | **一页融合正文** |
| 一篇 700 页文档大概打几次 | **约 700 次** |
| 提示词会变吗 | 基本不变（系统提示词固定） |
| 每页变化的是什么 | 只有「用户正文」那一段 |

## 3. 每次 LLM 包里到底有什么

每次调用，本质就两段文字：

### A. 系统提示词（固定，来自提示词表 id=2）

作用：告诉模型怎么抽、抽什么类、怎么写关系。

真实内容如下：

```text
你是护肤品牌企业知识库的实体抽取专家。从文档内容中抽取实体和关系,每个实体当场定类。

【实体名铁律——只取主体,剥离修饰】
- 实体名 = 能独立指代一个事物的最短主体词,不是一句话。
- 剥掉所有修饰:海报/宣传/培训/方案/介绍/说明/展示 等定语,活动/项目/资料 等后缀,数字编号,标点。
  例:「华世王镞2024春季新品培训方案」→ 主体是「华世王镞」(品牌);培训方案是文档类型不是实体。
  例:「舒颜莹润啫喱30ml」→ 实体「舒颜莹润啫喱」(产品),规格「30ml」单独一条(规格)。
  例:「烟酰胺美白精华」→「烟酰胺」(成分)、「美白」(功效)、「精华」(品类) 三条,不要合成一句。
- 抽不出干净主体的(整句是宣传语/描述句)→ 归 noise,别硬抽。

【当场定类——只能用下面 18 类的 type_name,不许自造】
成分 原料 功效 品类 产品 品牌 系列 规格 肤质 人物 组织 地点 事件 时间 技术标准 视觉素材 营销内容 噪音
- 拿不准归哪类 → 填「噪音」并说明,别瞎归(宁可 noise 不要错类)。
- 图片像素数据(RGB/亮度/边缘密度/色彩模式)、文件格式(JPEG/PNG)、纯数字、单字 → 一律「噪音」。

【关系——只用下面 9 种谓词】
拥有 属于 包含 相关 引用 参与 位于 产生 领导
- 品牌与产品:品牌「拥有」产品;产品「属于」品牌。方向别反。
- 每条关系必须有原文依据,没依据不要编。

返回 JSON(不要 markdown 代码块标记):
{
  "entities": [
    {"name": "主体词", "type_name": "18类之一", "description": "一句话依据", "confidence": 0.0到1.0}
  ],
  "relationships": [
    {"source": "主体A", "target": "主体B", "predicate": "9种谓词之一", "evidence": "原文片段"}
  ]
}
```

### B. 用户正文（每页不同）

代码固定包一层：

```text
请提取以下内容的实体和关系：

（下面接这一页的融合正文，最多截断到约 6000 字）
```

### 4.3 这一次调用说明了什么

- 输入其实很短（封面页）
- 输出已经有 9 个实体 + 8 条关系
- 主体抽取大体可用：产品、公司、评估人、日期都出来了
- 也有可讨论点：
  - 「安全评估」是否该作为事件实体
  - 地址整段是否应拆细
  - 页码被正确标成噪音（这点是好的）

## 5. 再强调一次：为什么会变成“每页一次”

当前实现就是：

```text
第1页正文  → 打 1 次 LLM
第2页正文  → 打 1 次 LLM
第3页正文  → 打 1 次 LLM
...
第N页正文  → 打 1 次 LLM
```

所以：

- 30 页文档 ≈ 30 次
- 700 页文档 ≈ 700 次

这不是“融合页还在比三路”，  
而是 **抽实体节点自己按页循环调用模型**。

---

## 6. 你后面改算法时，可以直接盯这 3 个问题

1. **输入粒度**  
   现在是“一页一段正文”。要不要改成“整篇关键页合并后一次抽”？

2. **提示词目标**  
   现在要求每页都抽实体+关系+18类定类。  
   要不要改成“只抽文档主体，不要求每页完整图谱”？

3. **输出用途**  
   现在每页结果先攒起来，最后统一落库归并。  
   如果改成文档级抽取，归并压力会小很多。

---

## 7. 一句话总结

当前抽实体节点，每次发给 LLM 的东西很简单：

```text
【固定系统提示词】
+ 
【请提取以下内容的实体和关系：】
+ 
【这一页的融合正文】
```

模型每次返回：

```text
实体列表（名字/类型/依据/置信度）
+
关系列表（谁-关系-谁/证据）
```

**单位是页，不是文档。**  
所以你会看到“页数 ≈ 调用次数”。

# 附录：如果按“规则优先”优化，我会怎么做（纯推算，不写代码）

> 这一节不是当前线上实现，是**优化推演**。  
> 用真实文档：`美容院服务礼仪`（约 40 页，正文总共大约六千多字）。  
> 只讲文字流程 + 实例，方便你判断对不对。

## 甲、先立原则

1. **主体按整篇文档认，不按页重复认。**  
2. **高置信走规则，低置信才问模型。**  
3. **模型是补洞，不是主发动机。**  
4. **每一步都要落盘留痕，不能等全跑完才写。**  
5. **同一主体第二次出现，默认并进已有主体，不重新开户。**

---

## 乙、优化后的总流程（文本图）

```text
拿到一篇文档的全部融合页正文
        │
        ▼
第一步：拼成“整篇可读正文”
（页与页之间用分隔线，保留页号，方便回挂证据）
        │
        ▼
第二步：规则层扫一遍
  - 标题/岗位词
  - 重复出现的主体词
  - 已有词典命中
  - 明显噪音（页码、纯口号、过短句）
        │
        ▼
第三步：给每个候选打置信度
  高：直接入“已确认主体库”
  中：入“待复核”
  低：入“问模型队列”
        │
        ▼
第四步：判断要不要问模型
  若“低置信片段”为空 → 完全不问模型
  若只有少量难句 → 只把难句打包问一次
  若全文主体都乱 → 才考虑整篇问一次
        │
        ▼
第五步：模型只返回“补洞结果”
（不是把全文再抽一遍）
        │
        ▼
第六步：合并
  规则结果 + 模型补洞结果
  同一主体合并
  关系去重
        │
        ▼
第七步：落库 / 归并 / 建图
（每一步都先写留痕，再进下一步）
```

### 3）规则怎么判断“同一个实体不要重复开户”

例子：

```text
第1页出现：美容院
第4页又出现：星级美容院
第5页又出现：美容院店面形象
```

规则判断：

1. 核心词都是“美容院”  
2. “星级”是修饰，不是新主体  
3. “店面形象”是属性描述，不是新主体  

所以结果应是：

```text
只保留一个主体：美容院
证据页：1、4、5
出现次数：多次
```

**这里根本不需要模型。**

再例：

```text
第4页：美容师
第6页：美容师正确的站姿
后面每页都在讲美容师动作
```

规则：

```text
主体仍是：美容师
后面只追加证据和属性
不新建“美容师正确的站姿”这种长名实体
```

### 4）什么时候才需要模型

对这篇文档，我会把全文分成三类片段：

#### 甲类：规则已吃掉（不问模型）
- 岗位名单：店主管、迎宾、前台、顾问、美容师  
- 标题主体：美容院、服务礼仪  
- 明确噪音：页码、重复口号、空姐类比  

#### 乙类：名字确定，类型拿不准（可问一次，也可规则兜底）
- “服务礼仪”到底算“营销内容”还是“技术标准”  
- “培训目标”是不是事件  

这类可以：
- 要么统一规则：培训材料里的规范类 → 营销内容/技术标准  
- 要么攒成**一个小问题包**问模型一次

#### 丙类：真正难、值得问模型
这篇其实很少。  
如果硬找，大概是：

```text
“创造品牌” 里的“品牌”是不是业务主体？
“星级美容院” 是否要和“美容院”分开？
```

即便问，也不是把 40 页全文再扔进去，而是只扔：

```text
难句1：……
难句2：……
已确认主体：美容院、美容师、前台……
请只裁决这些疑难点，不要重复已确认主体。
```

## 戊、置信度怎么分流（这是核心）

我给每个候选打三档：

### 高置信（直接过，不问模型）
满足任意两条以上：
- 标题命中
- 全文重复出现多次
- 命中岗位/品牌/产品词典
- 结构很干净（短主体词，无长修饰）
- 已在本篇“已确认主体库”里出现过

### 中置信（先落“待复核”，可稍后批量问）
- 能抽出词，但类型有两种可能  
- 出现次数少  
- 像主体，也像描述

### 低置信（必须问模型，或直接丢弃）
- 整句宣传语  
- 修饰很多，剥不干净  
- 和已有主体冲突  
- 只有一次出现，且上下文含糊

### 对这篇文档的推算结果

```text
规则直接确认：大约 6 到 10 个主体
需要模型裁决：0 到 3 个疑难点
需要模型调用次数：0 次或 1 次
绝不是 40 次
```

## 己、和当前错误做法对比

### 当前错误做法

```text
40 页
→ 每页问一次模型
→ 每页都可能抽出“美容院”“美容师”
→ 后面再归并
→ 又贵又慢又重复
```

### 优化后

```text
40 页先当一篇文档看
→ 规则扫出主体
→ 重复出现直接合并
→ 只把少数疑难点问模型
→ 0 次或 1 次模型调用
→ 每一步都留痕
```

### 费用感对比（粗算）

| 方式 | 这篇大约问模型几次 | 风险 |
|---|---:|---|
| 现在按页问 | 约 40 次 | 重复、浪费、一页挂了前面白忙 |

# 附录四：存量补齐 vs 新文档接入（本质同一套）

## 一、先分清两件事

| 场景 | 是什么 | 体量 |
|---|---|---|
| **现在这批** | 临时问题：老库已经有大量融合正文，但实体语义库存缺、乱、贵 | 全库/大批量补齐 |
| **完整算法** | 新文档入库后，按同一套规则当场枚举主体 | 单篇或小批量 |

**算法本质一致，只是体量不同。**

不是两套哲学：  
一套给老数据，一套给新数据。  
而是：

```text
同一套：
分词穷举 → 落盘复用 → 语义不清再批量定类

差别只在：
- 现在：对存量一口气补
- 以后：新文档进来就按同样规则跑
```

## 二、完整算法（新文档）

```text
新文档入库
  → 有了融合正文
  → 立刻分词 / 切词组
  → 枚举主体候选
  → 命中已有库存就复用
  → 新词/新组合落盘
  → 语义不清的进待定队列
  → 凑批后批量问模型定类
  → 写回类型、关系、证据
```

新文档不会“等全库重跑”，  
而是**边进边枚举**。

## 三、当前临时任务（存量）

```text
老库已有大量融合页
  → 缺的是：统一的主体库存 + 清晰语义
  → 所以现在要补：
     全局（或分批）分词穷举
     落盘复用
     把语义不清的打包成试卷批量定类
```

这是**补课**，不是永久形态。  
补完后，日常只跑“新文档路径”。

## 四、为什么说本质一致

| 步骤 | 新文档 | 存量补齐 |
|---|---|---|
| 分词穷举 | 对这一篇 | 对很多篇 |
| 落盘复用 | 写入/命中总库存 | 同样写入/命中总库存 |
| 批量定语义 | 这篇产生的待定题 | 全库积压的待定题 |
| 模型角色 | 补洞批改试卷 | 补洞批改试卷 |
| 差别 | 体量小 | 体量大 |

所以：

- **不是**“存量用全局算法，新文档又退回每页打模型”
- **而是**“库存算法是主算法；现在只是先把历史债补上”

## 五、一句话

```text
完整算法：新文档进来就枚举主体
当前工作：存量缺这块，先按同一规则补齐
两者本质相同，只是一次做一篇，还是一次补一批
```

# 附录五：每个词组都要有语义（不是“噪音=丢掉”）

## 原则

```text
词组必须有语义角色
        │
        ├─ 进主图谱 / 主召回     （graph_include = 是）
        └─ 不进主图谱，但仍保留  （graph_include = 否）
              用于：
              - 完整上下文
              - 共现关系
              - 自研召回算法
              - 后续语义组合
```

所以：

- **不是**：噪音 → 扔掉  
- **而是**：先给语义 → 再决定召回策略

## 语义角色例子

| 词组 | 语义角色 | 是否默认进主图谱 |
|---|---|---|
| 娇薇诗 | 业务类型:品牌 | 是 |
| 润颜精萃霜 | 业务类型:产品 | 是 |
| 正确的 | 修饰限定 | 否 |
| mg / kg | 计量数值 | 否 |
| 检验 / 报告 | 模板套话 | 否 |
| 的 / 和 / 以及 | 功能连接 | 否 |
| 某长词暂不清 | 待定主体/待定语义 | 先保留，再进试卷补标 |

## 为什么必须这样

如果“非主体”直接当噪音丢掉：

1. 句子结构碎掉  
2. 组合语义组不起来  
3. 共现召回缺上下文  
4. 后续算法无法用完整语言信号提精度  

正确做法：

```text
全量保留词组语义
→ 主体进图谱
→ 非主体留在库存与共现网
→ 召回时按策略取用
```
