## ZCode 配置

```text
Base URL: http://127.0.0.1:8317/v1
API Key: 123
Model: grok-4.5
Protocol: OpenAI Chat Completions
```

如果 ZCode 流式经常报“模型未返回任何内容”，优先改用稳定代理：

```text
Base URL: http://127.0.0.1:8318/zcode/v1
API Key: 123
Model: grok-4.5
Protocol: OpenAI Chat Completions
```

`zcode` 是智能体壳名，可以换成其它壳名，例如：

```text
http://127.0.0.1:8318/hermes/v1
http://127.0.0.1:8318/codex/v1
```

稳定代理会按壳名和请求体生成缓存 key。同一个壳、同一个请求如果中途重试，会优先返回上一次完整成功的结果。

稳定代理也会读取会话 ID，并转成 CPA 认可的 `X-Session-ID` / `Session-Id`：

```text
优先读取 header:
X-Session-Id
X-Conversation-Id
X-Chat-Id
X-Thread-Id
X-Agent-Session-Id
X-ZCode-Session-Id
X-Codex-Session-Id

其次读取 body / metadata:
session_id
conversation_id
chat_id
thread_id
task_id
run_id
```

同一个 `智能体壳 + 会话 ID` 会被映射成同一个 CPA session key。验证时 CPA 日志会显示 `session=header:...`，而不是 `session=msg:...`。

注意：`/v1/models` 需要带 `Authorization: Bearer 123`，否则会返回 `401`。如果 ZCode 的模型列表刷新不带 key，可能会显示异常，但聊天接口带 key 后仍可用。

## 探针

模型列表：

```bash
curl -sS \
  -H 'Authorization: Bearer 123' \
  http://127.0.0.1:8317/v1/models
```

非流式 chat：

```bash
curl -sS \
  -H 'Authorization: Bearer 123' \
  -H 'Content-Type: application/json' \
  http://127.0.0.1:8317/v1/chat/completions \
  -d '{"model":"grok-4.5","stream":false,"messages":[{"role":"user","content":"只回复 hi"}]}'
```

流式 chat：

```bash
curl -sS -N \
  -H 'Authorization: Bearer 123' \
  -H 'Content-Type: application/json' \
  http://127.0.0.1:8317/v1/chat/completions \
  -d '{"model":"grok-4.5","stream":true,"messages":[{"role":"user","content":"只回复 hi"}]}'
```

稳定代理流式 chat：

```bash
curl -sS -N \
  -H 'Authorization: Bearer 123' \
  -H 'Content-Type: application/json' \
  http://127.0.0.1:8318/zcode/v1/chat/completions \
  -d '{"model":"grok-4.5","stream":true,"messages":[{"role":"user","content":"只回复 hi"}]}'
```

正常非流式响应会包含：

```json
{
  "object": "chat.completion",
  "choices": [
    {
      "message": {
        "role": "assistant",
        "content": "hi",
        "reasoning_content": "..."
      }
    }
  ]
}
```

正常流式响应会先吐 `delta.reasoning_content`，最后才出现 `delta.content`。如果客户端只看前几个 chunk，或把纯 reasoning chunk 当成正文结束，可能误报“模型未返回任何内容”。

稳定代理会把上游请求改成非流式，等完整成功响应回来后缓存，再向客户端输出干净 SSE：

```text
data: {"choices":[{"delta":{"role":"assistant","content":"hi"}}]}
data: {"choices":[{"delta":{},"finish_reason":"stop"}]}
data: [DONE]
```

## 当前不稳定点

日志里已经观察到：

- 大多数 `/v1/chat/completions` 是 `200`
- 有一次请求约 5 分钟后返回 `500`
- 多次出现 `session-affinity: cache hit but auth unavailable, reselected`
- 有一次未带 key 的 `/v1/models` 返回 `401`

这说明 CPA 服务本身是活的，账号池重选也在工作；不稳定更像是两类问题：

1. 上游账号或会话偶发不可用，导致长等待后失败。
2. ZCode 对 Grok 的流式 reasoning chunk 兼容不完全，可能把有返回的请求判成空内容。

当前已加的稳定措施：

1. `cpa-pool-prune-bad.sh` 每 60 秒扫描 `stdout.log`，发现 `Suspended client ... payment_required` 就把账号从 `auths/active` 移到 `auths/depleted`。
2. 自动摘坏号累计达到 10 个时，调用 `cpa-clash-rotate-ip.py` 切换 CPA 专属 Clash 节点。
3. CPA 不再走系统 ClashX / 旧 probe 端口 `4783`，而是走独立 Clash 核心 `4883`。
4. 稳定代理按会话 ID 注入 CPA 认可的 `X-Session-ID` / `Session-Id`，让同一 ZCode 会话尽量固定到同一组路由。
5. `cpa-auth-health-report.py` 会统计“曾经成功、后来 403/payment_required”的账号，用于判断号质量是否在持续衰减。
6. 稳定代理会完整转发 `tool_calls`，包括“正文为空但只有工具调用”的合法响应，避免 ZCode 停在一句话。
7. CPA 返回 5xx 时，稳定代理会临时换一个 session key 自动重试一次；正常请求仍保持同会话固定号。

优先排查顺序：

1. ZCode 里确认 API Key 确实随 `/v1/models` 和 `/v1/chat/completions` 一起发送。
2. 如果 ZCode 可选，先关闭 streaming，验证非流式是否稳定。
3. 如果必须 streaming，可加本地兼容层：转发到 `8317`，过滤纯 `reasoning_content` chunk，只把 `delta.content` 交给客户端。
4. 对 5 分钟 `500`，继续观察 stdout 里对应 request id 前后的 auth 重选记录；必要时把频繁 unavailable 的账号摘到 `auths/depleted`。
