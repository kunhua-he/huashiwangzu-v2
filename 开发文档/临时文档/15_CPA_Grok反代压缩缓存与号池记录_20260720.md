# CPA Grok 反代压缩缓存与号池记录

日期：2026-07-20

## 背景

本轮主要围绕 CPA Grok 反代的稳定性、号池耗尽、上下文过大和中间件压缩缓存展开。

核心问题是：Zcode、Hermes 等 agent 每轮都会把完整上下文发给反代，导致单次请求体越来越大。即使账号可以继续补充，如果每个请求都直接把 400K、600K 甚至更大的上下文发给 Grok，很容易几轮就打穿账号额度。

用户明确希望：不改变 agent 自己发出的请求内容，而是在 CPA stable proxy 中间件层做压缩缓存。第一次请求将旧上下文压缩成高精度摘要，第二次同会话请求进来时，复用前缀摘要，只处理新增 delta，再把“摘要 + 最近原文窗口”发给模型。

## 主要结论

当前 CPA stable proxy 已经支持 rolling 压缩缓存。

实现目标：

- agent 请求原样进入中间件，Zcode/Hermes 侧不用改。
- 中间件按 `agent + session_id` 维护会话级摘要状态。
- 同一个会话的历史前缀只压缩一次。
- 后续请求只对新增旧消息做增量总结。
- 发送给 Grok 的请求变成“rolling summary + 最近 18 条原文消息”。
- 最近原文消息优先级高于摘要，避免摘要和最新上下文冲突。

## 压缩流程

第一次同会话大请求：

1. 解析 agent 和 session id。
2. 提取旧消息：`messages[:-18]`。
3. 保留最近 18 条原文消息。
4. 对旧消息做 LLM 摘要。
5. 保存旧消息 hash 列表和摘要到 session summary store。
6. 转发给 Grok 的请求体变成“摘要 + 最近 18 条原文”。

后续同会话请求：

1. 再次计算旧消息 hash。
2. 和上一次保存的 hash 列表做前缀匹配。
3. 如果前缀一致，只拿新增旧消息作为 delta。
4. 让 LLM 把 delta 合并进已有摘要。
5. 更新 session summary store。
6. 继续只转发“更新后的摘要 + 最近 18 条原文”。

如果前缀不一致：

- 自动进入 `mode=rebuild`，重新从当前旧消息构建摘要。

如果 rolling 压缩失败：

- 继续走原来的 650KB 触发式 LLM 压缩。
- 再失败则走规则压缩兜底。
