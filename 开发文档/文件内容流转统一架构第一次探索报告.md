# 文件内容流转统一架构第一次探索报告

## 1. 当前真实流程

### 1.1 用户上传/拖入

```
用户拖入/上传
  │
  ▼
POST /api/files/upload           ← backend/app/routers/file_transfer.py
  │                                 流式写入临时文件，同时计算 MD5
  ▼
file_upload_service.upload_file_from_path()   ← backend/app/services/file_upload_service.py
  │
  ├─ MD5 去重检测 (framework_file_items.md5_hash)
  ├─ 内容寻址存储: uploads/{md5[:2]}/{md5[2:4]}/{md5}.{ext}
  ├─ 同 md5 文件已存在 → ref_count+1，复用 storage_path
  ├─ 同名冲突 → 409 (A file with the same name already exists)
  └─ 回包: {id, name, extension, size, mime_type, deduplicated}
       │
       ▼
  emit_module_event("file.uploaded", {file_id})  ← 事件总线
       │
       ▼  (异步，不阻塞上传)
  knowledge 模块 _on_file_uploaded handler
       → _cap_ingest → POST /knowledge/documents
                        → 创建 kb_document 记录
                        → 入队 kb_pipeline 任务 (parse + collect_raw + chunk + fusion)
```

**关键事实**：
- 入口接口：`POST /api/files/upload`（框架层）
- 原始文件记录：`framework_file_items` 表
- 物理文件存储：`uploads/{md5[:2]}/{md5[2:4]}/{md5}.{ext}`（内容寻址路径）
- 桌面壳显示：`framework_file_items.id` + `name` + `extension`（通过 `/api/files/list`）
- 是否自动触发知识库 ingest：是，通过事件总线 `file.uploaded` → `_on_file_uploaded` 自动触发
- 知识库 ingest 是同步还是异步：**异步**，消息入队 framework_system_task_queues 后由后台 worker 处理
- 知识库解析时调用格式模块：是，通过 `parse_document()` → `call_capability(module, action, {file_id})` 调用 parser 模块
- 同一个 xlsx/docx/pptx 是否被多模块重复解析：**是的（见第 3 节）**

### 1.2 知识库 Ingest 管线

```
kb_document 创建
  │
  ▼
full_pipeline (异步后台)
  │
  ├─ Step 1: parse_document() → DocumentIr
  │    调用 parser 模块 capability (pdf-parser:parse, docx-parser:parse, 等)
  │    → 得到统一 DocumentIr (blocks + resources)
  │
  ├─ Step 2: chunk_and_embed()
  │    分块 → kb_chunks + 向量化
  │
  ├─ Step 3: collect_raw_data()  ← 三/二/一轮并行采集
  │    ├─ Round 1 (text):  调用 parser 提取文本 → kb_raw_data {source_type:"text"}
  │    ├─ Round 2 (OCR):   渲染页面 → pytesseract → kb_raw_data {source_type:"ocr"}
  │    └─ Round 3 (VLM):   渲染页面 → LLM 视觉 → kb_raw_data {source_type:"vision"}
  │    (PDF 走三轮，图片走两轮，Office/文本走一轮)
  │
  ├─ Step 4: fuse_page_data()
  │    多轮融合 → kb_page_fusions (fused_text + body_json + conflicts_json)
  │
  ├─ Step 5: generate_profile()
  │    文档画像 → kb_document_profiles (subject, summary, key_entities)
  │
  ├─ Step 6: entity extraction → kb_entity_dictionary + kb_graph_nodes/edges
  │
  └─ Step 7: cross-file relation → kb_file_relations
```

### 1.3 格式解析

```
parse_document(file_id, extension)
  │
  ├─ get_parser_for_format(extension)
  │   FORMAT_PARSER_MAP = {
  │     "pdf":   ("pdf-parser", "parse"),
  │     "docx":  ("docx-parser", "parse"),
  │     "pptx":  ("pptx-parser", "parse"),
  │     "xlsx":  ("xlsx-parser", "parse"),
  │     "csv":   ("csv-parser", "parse"),
  │     "txt":   ("text-parser", "parse"),
  │     "md":    ("markdown-parser", "parse"),
  │     "json":  ("structured-parser", "parse"),
  │     "yaml":  ("structured-parser", "parse"),
  │     "eml":   ("email-parser", "parse"),
  │     "msg":   ("email-parser", "parse"),
  │   }
  │   image formats → ("image-vision", "describe")
  │
  └─ call_capability(module_key, action, {file_id})
       → /api/modules/call 跨模块调用
       → parser 模块返回 {format, blocks, resources}
       → from_legacy_blocks() → DocumentIr
```

**注意**：parser 模块通过 framework capability 调用，knowledge **不直接 import** parser 模块代码。模块边界遵守良好。

### 1.4 Agent 读取

Agent 目前通过 `knowledge:search` capability 检索知识库，读的是 `kb_chunks` 和 `kb_page_fusions`（融合层）。没有直接读原始 JSON 包的 Agent 工具。

### 1.5 编辑

- **Office JSON Package**：Framework 层 `backend/app/services/office/` 有 `DocxService.parse()`/`PptxService.parse()`/`ExcelService.parse()`，生成 JSON → 存 `framework_file_json_packages` + `framework_file_json_versions`
- **Excel Engine**：独立模块有 `excel_workbooks`/`excel_sheets`/`excel_cells` 等表，独立编辑能力通过 `excel-engine:update_range`、`excel-engine:append_rows` 等 capability 暴露
- **office-gen**：生成 Office 文件的模块，通过 `office-gen:docx/xlsx/pptx` capability 直接生成物理文件

### 1.6 导出/下载

```
GET /api/files/download/{file_id}  ← 框架统一
  │
  ├─ check_file_access(file_id, user_id)
  ├─ _resolve_storage_path(file)
  └─ FileResponse(物理文件)

office-gen 导出流程：
  office-gen:docx/xlsx/pptx → 生成物理文件 → POST /api/files/upload
  → 返回新 file_id

artifact 导出：
  create_artifact → upload_file → framework_file_items
  download_via artifact.file_id
```

## 3. 当前重复和冲突点

### P1: Office 格式解析有三套，且都是真实现

| 解析路径 | 代码位置 | 用途 |
|---------|---------|------|
| 1. Framework Office Services | `backend/app/services/office/docx_service.py` | office-gen 导出时重编译用 |
| 2. Parser 模块 | `modules/{docx,pptx,xlsx,…}-parser/backend/router.py` | knowledge 调用 parse_document |
| 3. Excel Engine 自己 | `modules/excel-engine/backend/router.py` 内自带 `self._open_xlsx()` | Excel 编辑器独立解析 |

**问题**：同一个 xlsx 文件上传后，会被 knowledge（通过 xlsx-parser）和 excel-engine（直接 openpyxl）各解析一次，未来可能还有 framework Office Service 再解析一次。三套代码互不认识，输出格式不同，没有复用。

### P2: Framework Office Services 和 Parser 模块是重复的

`backend/app/services/office/` 能 parse/export docx/pptx/xlsx（通过 python-docx/python-pptx/openpyxl）。Parser 模块也能 parse 同样的格式（也是通过同样的库）。两者的 parse 输出格式不同：
- Framework Service 输出：`{manifest, content/paragraphs/tables}` 或 `{version, metadata, sheets}`
- Parser 模块输出：`{format, blocks, resources}` (标准化 DocumentIr)

且 `office-gen:export_to_artifact` 用了 Framework Service 的 export，knowledge 用了 Parser 模块的 parse。**同格式的两个解析实现应该合并**。

### P3: 没有统一的内容包结构

- Framework `FileJsonPackage` 存的是 JSON，但 `create_package()` 创建时 content 是 `{"content": {}}`（空壳），真正的内容在 `framework_file_json_versions.json_content` 里
- Parser 模块返回 `{format, blocks, resources}`（通过 `from_legacy_blocks` 转成 `DocumentIr`）
- Knowledge 把解析结果存到 `kb_raw_data`/`kb_chunks`/`kb_page_fusions`，而不是保留原始解析 JSON
- 没有标准的包结构、资源引用规范、版本号一致

### P4: 资源/图片没有统一存储

- PPTX parse 时虽然识别了图片 shape（`resource_id: shape.image.content_type`），但实际没有提取图片到独立存储
- DOCX parse 完全忽略图片
- `framework_file_assets` 表存在，但用于 asset 发布跟踪，不是媒体资源库
- 没有按 hash 去重的资源表（`framework_resources`），同一张图片出现在多个文件里会存多份
- Image-vision module 通过 `file_id` 操作，但知识库里的 page 图片是渲染后的临时字节，没有走资源表

### P5: Excel 有独立版本体系脱离框架

`excel_versions` 表有自己的 version 管理，与 `framework_artifact_versions` 或 `framework_file_json_versions` 没有关联。Excel Engine 的 `publish_to_desktop` 创建物理文件但**不创建 artifact**。

### P6: Artifact 和 File 职责重叠

- `framework_artifacts` 可以 linked to `file_id`（file 模式）、或存储 `content_json`（db 模式）
- `framework_file_items` 已经是物理文件的记录
- `framework_artifacts.file_id` 和 `framework_file_items` 有重叠：同一个物理文件可能在两个表里都有记录
- `desktop-tools:publish_artifact` 操作：下载 artifact 内容 → 上传为新文件 → 返回新 file_id，这个过程创建了新的 `framework_file_items` 记录
- 桌面显示的是 `framework_file_items`，不是 artifact

### P7: 版本对象不统一

| 版本表 | 归属 | 快照内容 |
|--------|------|---------|
| `framework_artifact_versions` | 框架 | snapshot_json + file_id + storage_path |
| `framework_file_json_versions` | 框架 | json_content (纯文本) |
| `excel_versions` | excel-engine | snapshot_json + storage_path + file_size |
| `kb_documents` 有 `parse_status/raw_status/fusion_status` | knowledge | 状态字段，无版本号 |

**冲突**：artifact 版本的 `file_id` 指向物理文件版本；excel 版本 `file_id` 也指向物理文件；但两者的版本号、快照格式、API 都不兼容。

### P8: office-gen 直接生成物理文件跳过中间层

`office-gen:docx/xlsx/pptx` 从外部传入 `content/sheets/slides` 直接生成 Office 文件并上传。**没有经过 JSON package / artifact / content package**。生成的物理文件可以直接下载，但无法后续被 Agent 结构化编辑。

### P9: Knowledge 搜索返回 chunk/fusion 层，Agent 无法获取结构化内容

Agent 通过 `knowledge:search` 搜索，返回的是文本块和融合文本。如果 Agent 需要"修改第 3 张幻灯片第 2 段文字"，目前没有工具能：
1. 获取原始结构化内容（幻灯片结构、段落 ID）
2. 定位并修改特定元素
3. 重新编译为 PPTX

Agent 面对的是**文本碎片**而非 **结构化内容包**。

```
  │
  ▼
┌─────────────────────────────────────────────────┐
│  Step 1: 框架登记原始文件                         │

│  → emit file.uploaded                            │
└─────────────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────────────┐
│  Step 2: 内容预处理管线 (框架级)                   │
│  ┌─────────────────────────────────────────────┐ │

│  │  ├─ 按 format adapter 调用 parser 模块       │ │
│  │  ├─ parser 模块返回 {manifest, blocks,       │ │
│  │  │   resources}                              │ │
│  │  ├─ 图片资源提取 → framework_resources       │ │

  ├────────────────────────────────────┐

```

### 与用户提议流程的差异

| 用户提议 | 建议调整 | 原因 |
|---------|---------|------|
| 框架登记 → 分析/预处理 → 按格式调用解析 | ✅ 一致 | 当前已做，但需要标准化输出 |
| 生成统一 JSON + 资源包 | ✅ 一致 | 当前不存在，需要新建 |
| 知识库做三层基础原始数据 | ⚠️ 微调 | 知识库应该优先消费 content package，而非自己从头解析；Round2 OCR 和 Round3 VLM 仍然是知识库的增值层 |
| 知识库做第四层唯一值融合 | ✅ 一致 | kb_page_fusions 已经是这个角色 |
| Agent 后续只读取结构化 JSON | ✅ 一致 | 但需要 content package 工具层，Agent 不能直接读 table |
| Agent 修改时操作 JSON | ✅ 一致 | 需要 content:update_* 工具 |
| 用户下载时再编译成物理文件 | ✅ 一致 | 需要 export adapter 标准化 |

## 5. 推荐数据模型

### 5.1 框架级表（新增）

```sql
-- 内容包（统一结构化内容）
framework_content_packages
  id            BIGSERIAL PRIMARY KEY
  source_file_id INTEGER NOT NULL REFERENCES framework_file_items(id)
  package_type  VARCHAR(32) NOT NULL  -- 'document' / 'spreadsheet' / 'presentation' / 'image' / 'generic'
  package_version VARCHAR(16) NOT NULL DEFAULT '1.0'
  manifest_json JSONB         -- 包级别元数据（标题、页数、作者等）
  current_version_id INTEGER REFERENCES framework_content_package_versions(id)
  status        VARCHAR(32) NOT NULL DEFAULT 'pending'  -- pending / parsed / failed
  created_at    TIMESTAMPTZ
  updated_at    TIMESTAMPTZ

-- 内容包版本（快照）
framework_content_package_versions
  id            BIGSERIAL PRIMARY KEY
  package_id    INTEGER NOT NULL REFERENCES framework_content_packages(id)
  version_no    INTEGER NOT NULL
  content_json  JSONB NOT NULL   -- blocks + document structure（不含资源）
  summary       TEXT
  creator_id    INTEGER NOT NULL
  created_at    TIMESTAMPTZ

-- 资源存储（MD5 内容寻址，媒体唯一存储）
framework_resources
  id            BIGSERIAL PRIMARY KEY
  hash          VARCHAR(64) NOT NULL UNIQUE  -- sha256
  hash_algorithm VARCHAR(16) NOT NULL DEFAULT 'sha256'
  resource_type VARCHAR(32) NOT NULL  -- image / audio / video / binary
  mime_type     VARCHAR(128) NOT NULL
  storage_path  VARCHAR(1024) NOT NULL  -- uploads/res/{hash[:2]}/{hash[2:4]}/{hash}.{ext}
  file_size     BIGINT NOT NULL
  width         INTEGER   -- for images
  height        INTEGER   -- for images
  duration_ms   INTEGER   -- for audio/video
  description   TEXT      -- image-vision 写入的视觉描述
  ocr_text      TEXT      -- OCR 结果
  vlm_metadata  JSONB     -- VLM 理解结果
  ref_count     INTEGER NOT NULL DEFAULT 1
  created_at    TIMESTAMPTZ

-- 资源引用（包 ←→ 资源）
framework_resource_refs
  id            BIGSERIAL PRIMARY KEY
  package_id    INTEGER NOT NULL REFERENCES framework_content_packages(id)
  resource_id   INTEGER NOT NULL REFERENCES framework_resources(id)
  usage_type    VARCHAR(32) NOT NULL  -- embedded / referenced / thumbnail
  page          INTEGER   -- 所在页/幻灯片
  coordinates   JSONB     -- 图片在页中的坐标
  usage_hints   TEXT[]    -- 描述该资源在文档中的用途
  created_at    TIMESTAMPTZ
  UNIQUE(package_id, resource_id, usage_type, page)
```

## 6. 推荐能力设计

### 6.1 框架级能力

| 能力 | 说明 | 优先级 |
|------|------|--------|
| `content:pipeline` | 文件上传后自动/手动触发预处理管道 | P0 |
| `content:get_package` | 获取包的 manifest + blocks（无资源） | P0 |
| `content:get_full_package` | 获取完整包含资源元数据 | P0 |
| `content:list_blocks` | 列出所有 block（按类型/页过滤） | P0 |
| `content:get_block` | 获取单个 block | P0 |
| `content:update_block` | 更新单个 block（文本/样式） | P0 |
| `content:replace_text` | 全文查找替换 | P1 |
| `content:append_blocks` | 追加 block（表格行/幻灯片/段落） | P1 |
| `content:list_resources` | 列出包内资源 | P0 |
| `content:get_resource` | 获取资源描述（含 OCR/VLM 结果） | P0 |
| `content:export` | 编译 Content Package → 物理文件 | P0 |
| `content:publish` | 发布到桌面（创建 artifact + file） | P1 |
| `content:list_versions` | 包版本历史 | P0 |
| `content:restore_version` | 回滚到指定版本 | P1 |

### 6.2 模块级保留能力

| 能力 | 模块 | 说明 |
|------|------|------|
| `pdf-parser:parse` | pdf-parser | PDF 文本+版面解析 |
| `docx-parser:parse` | docx-parser | DOCX 结构化解析（合并 framework DocxService） |
| `xlsx-parser:parse` | xlsx-parser | XLSX 结构化解析 |
| `pptx-parser:parse` | pptx-parser | PPTX 结构化解析（含图片提取） |
| `image-vision:describe` | image-vision | 图片视觉描述 |
| `knowledge:search` | knowledge | 语义检索（读融合层） |
| `knowledge:get_page_fusion` | knowledge | 页级融合数据 |
| `knowledge:get_graph_context` | knowledge | 图谱上下文 |
| `excel-engine:update_range` | excel-engine | Excel 实时编辑器操作 |
| `office-gen:docx/xlsx/pptx` | office-gen | 从原始材料生成 Office 文件（保留，但 export 路径改为走 content package） |

### 6.3 Agent 面向工具层

Agent 不应该面对 6.1 的全部框架能力，也不应该面对 6.2 的模块能力。推荐 Agent 暴露 3 类工具：

**第一类：内容浏览（稳定，所有文件类型通用）**
```
content:list_packages        # 列出工作区所有内容包
content:get_package          # 获取包结构（manifest + block 概览）
content:query_blocks         # 按类型/页/关键词查询 block（支持 SQL-like 过滤）
content:read_resource        # 读取资源描述/文本
```

**第二类：内容编辑（类型感知，按 block 操作）**
```
content:update_blocks        # 通用 block 修改
content:replace_text         # 全文替换（跨 block）
content:append_rows          # 表格专用（识别表格 block）
content:append_slides        # 幻灯片专用
content:restructure          # 重组（移动/删除/合并 blocks）
```

**第三类：生命周期**
```
content:export               # 编译为物理文件 + 下载
content:publish              # 发布（到桌面/分享）
content:list_versions        # 版本历史
content:restore_version      # 回滚
```

**关键设计原则**：
- Agent 不直接操作 `excel-engine:update_range` 或 `office-gen:xlsx`
- Agent 操作 Content Package，底层 export adapter 调用模块能力
- Excel 表格操作 = `content:update_blocks`（识别 block_type=table）+ `content:append_rows`
- PPT/DOCX 统一为 block 操作，不需要专用 PPT/DOCX 工具
- 图片资源：Agent 读 `resource.description`（image-vision 写入），不需要直接调 image-vision
- 样式 JSON 默认不返回给 Agent，仅在 Agent 显式 request `include_styles=true` 时携带

## 7. 各文件类型建议

### 7.1 xlsx

| 物理文件 | `framework_file_items` | 原始上传 + 最终导出 |

## 8. V2 里"文件内容"的唯一真相源

### 核心答案

**不同文件类型有不同的 canonical source，但统一入口是 Content Package：**

| 文件类型 | 唯一真相源（canonical source） | 原因 |
|---------|-------------------------------|------|
| **xlsx** | Content Package (blocks) + `excel_workbooks` (编辑态) | 两个不可互相替代：Agent 操作 block，UI 操作 cell 级状态 |
| **docx** | Content Package | 线性文档，block 操作全覆盖 |
| **pptx** | Content Package | 幻灯片结构，block 操作全覆盖 |
| **pdf** | Content Package（只读）+ `kb_page_fusions`（语义层） | PDF 不可逆编辑 |
| **image** | `framework_resources` | 图片本身 + VLM 描述 |
| **audio/video** | `framework_resources` | 媒体文件 + 转写 |
| **txt/md/json** | Content Package 或直接 `framework_file_items` | 纯文本不需要中间层 |

**一句话答案**：**Content Package 是所有结构化内容的统一入口和编辑源**，但其内部 block 结构根据文件类型不同；真正的唯一真相源需要两级看待：
- **物理层**（文件是什么）：`framework_file_items` + `framework_resources`
- **逻辑层**（文件内容是什么）：`framework_content_packages` + `framework_resource_refs`

> 原始物理文件**不是**逻辑唯一真相源（同一文件可以被不同 parser 产出不同的 blocks）；  
> Artifact**不是**唯一真相源（它是生命周期包装，不是内容本体）；  
> Knowledge**不是**唯一真相源（它是检索/融合层，不可写）；  
> 各格式模块自己的结构化表**不是**唯一真相源（分散、不互通）；  
> **Content Package 才是逻辑层的唯一真相源**。

## 9. 迁移策略

### 9.1 短期（当前架构可做的事，不破坏现有）

1. **标准化 Parser 模块输出**：所有 parser 模块返回 `{format, blocks, resources}` 格式（当前大部分已做，检查一致性）
2. **统一 Framework Office Services → 合并到 Parser 模块**：将 `backend/app/services/office/` 的 parse/export 能力迁移到对应的 parser 模块（`docx-parser`, `xlsx-parser`, `pptx-parser`），避免框架层再有格式解析代码
3. **为 knowledge 管线增加 content package 落盘**：`collect_raw_data` 完成后，将 parse_document 的结果存为 content package（新表），而不是只在 raw_data 里保留文本碎片

### 9.2 中期（新增框架能力，不破坏现有）

4. **创建 `framework_content_packages` + `framework_resources` 表**
5. **创建 `content:pipeline` 框架能力**：文件上传事件触发 content pipeline → 调用 parser → 存 content package + 资源
6. **修改 `kb_documents` 增加 `content_package_id` 引用**：知识库消费 content package，不再独立 parse
7. **PPTX/DOCX Parser 提取资源**：parser 模块从 Office zip 中提取图片 → 存 `framework_resources`
8. **创建 Agent 统一工具层**：`content:list_packages` / `content:get_package` / `content:update_blocks` 等

### 9.3 长期（收口，可废弃旧路径）

9. **废弃 `framework_file_json_packages` + `framework_file_json_versions`**：合并到 `framework_content_packages`
10. **废弃 `framework_file_assets`**：合并到 `framework_resource_refs`
11. **`office-gen:export_to_artifact` 改为走 content package → export adapter**
12. **Excel Engine 的 `publish_to_desktop` 改为走 content package → export adapter + artifact**

## 10. 风险和优先级

### P0（必须有，否则后续无法收口）

| 风险 | 描述 |
|------|------|
| Parser 模块输出格式不统一 | 当前 DocumentIr 格式有 `from_legacy_blocks`，但部分 parser 模块可能返回不同格式 |
| Framework Office Services 与 Parser 模块是平行重复 | 需要合并前确认哪套更完整 |
| 没有 `content:pipeline` 管线 | 文件上传后没有自动产生 content package |

### P1（应该做，否则 Agent 无法统一操作文件）

| 风险 | 描述 |
|------|------|
| 没有 `framework_resources` 表 | 图片/媒体无法去重存储，Office 内嵌资源无法引用 |
| Knowledge 管线不读 content package | 知识库仍然独立 parse，失去统一入口 |
| Agent 没有 content package 工具 | Agent 只能通过 knowledge search 或 file download 操作文件 |
| Excel 编辑态 (`excel_workbooks`) 与 content package 的同步策略 | 用户通过 Excel 编辑器修改后，content package 不会自动更新 |

### P2（做更好，不做也可过渡）

| 风险 | 描述 |
|------|------|
| Artifact 与 File 的重叠 | 短期不影响功能，但认知成本高 |
| `framework_file_json_packages` 和 `framework_file_json_versions` 是死代码 | 已有表但内容为空，确认是否可以立即废弃 |
| 版本系统不统一 | 当前各有各的版本，但互相不冲突 |
| `framework_file_assets` 的用途模糊 | desktop-tools 的 publish 用，但和 artifact 的 publish 重复 |
