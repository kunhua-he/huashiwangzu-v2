# 生产级底层审计报告（2026-06-18）

> 审计对象：华世王镞 V2 框架底座（Vue3 + FastAPI + PostgreSQL）
> 审计前提：局域网内部系统，约 30 人。重稳定性、数据一致、不崩、能扛模块接入；爆破类安全可适当放宽。
> 基线核对：后端 `pytest` 71 项全过；前端 `vue-tsc --noEmit` 0 错、`npm run build` 成功。
> 结论先行：**地基方向对、owner 隔离做得扎实，但还没到生产级。** 有 1 个 P0、3 个 P1 必须在接模块前处理。

---

## 总评

| 维度 | 评价 |
|------|------|
| 鉴权/授权隔离 | 良好。文件/文件夹/桌面状态/通知读状态全部按 `owner_id` 过滤，跨用户操作返回 403/404，几乎没漏。仅通知“模块发公告”有一处授权松。 |
| 异常契约 | 良好。统一 `{success,data,error}`，`AppException` 体系 + 全局兜底处理器齐全。 |
| 模型网关/SSE | 较好。SSE 不持有 DB 连接、httpx 带超时和退避重试、连接上下文托管，历史“SSE 死连接耗尽进程池”的坑这一版规避了。 |
| 前端健壮性 | 良好。监听器/ResizeObserver/定时器基本都在 `onUnmounted` 清理；全仓几乎无 `any`/`@ts-ignore`。 |
| **后台任务可靠性** | **不及格。框架任务队列没有任何消费者，提交即永久 pending。** |
| **资源边界** | **不及格。文件上传无大小限制且整文件读进内存，知识库一接入大文件就会 OOM。** |
| 启动时序/迁移 | 有隐患。坏 manifest 直接拖垮整个后端启动；`create_all` 与 Alembic 双schema源，迁移形同虚设。 |
| 去重/引用计数 | 设计不自洽。`ref_count` 字段写了但删除时根本不读它；复制功能破坏去重，并发删除会泄漏磁盘文件。 |

**是否达到生产级：否。** 修完下方「接模块前必修清单」后，框架可达到承载 AI 助手 / 知识库的水平。

## P0 致命（接模块前必修）

### P0-1 框架任务队列没有消费者，任务永久卡在 pending

**现象**：`/api/tasks/submit` 把任务写入 `framework_system_tasks`（`SystemTaskQueue`），status=`pending`。全仓没有任何进程/协程/调度器消费这张表。`worker/status` 端点会显示 `oldest_waiting_seconds` 无限增长。

**证据**：
- 提交端点：[tasks.py:103](backend/app/routers/tasks.py:103) 只写库，无投递。
- 全仓检索 `SystemTaskQueue` 的所有引用只有 router 读列表 / dashboard 统计 / 改状态，**没有任何地方把 pending 拉出来执行**（无 `asyncio.create_task` 循环、无 apscheduler/celery、`main.py` 的 lifespan 也没起 worker）。
- 唯一存在的 worker 是 `scripts/worker/knowledge_worker_runtime.py`，但它：①消费的是 `KnowledgeTask` 表，**不是** `SystemTaskQueue`；②`from app.models.knowledge import KnowledgeTask` —— 该模块在 V2 已删除，实测 `ModuleNotFoundError`，根本无法 import。即这是 V1 残留死代码，不是框架 worker。见 [knowledge_worker_runtime.py:9](backend/scripts/worker/knowledge_worker_runtime.py:9)。

**影响**：底层文档把“队列、定时任务、worker”列为已交付平台能力，但实际是个只进不出的黑洞。知识库模块的解析/向量化/分层处理必然要走异步任务——一接入，所有任务全部永久 pending，模块表现为“提交后永远不动”。这正是审计定义里的 P0（任务永不执行）。

**修复建议**：
1. 明确框架任务队列的事实标准：要么以 `SystemTaskQueue` 为唯一队列表，要么直接采用 `knowledge_worker_runtime.py` 里那套**已经写得不错的租约模型**（`lease_until` + `heartbeat` + 过期任务回收，见 [knowledge_worker_runtime.py:41](backend/scripts/worker/knowledge_worker_runtime.py:41)），把它从 V1 死引用迁成框架级 worker，消费 `SystemTaskQueue`。
2. worker 取任务的 `SELECT … LIMIT 1` + `UPDATE` 之间要加 `FOR UPDATE SKIP LOCKED`，否则多 worker 会抢同一条（当前 `_first_available_task`→`_lease_task` 之间无行锁）。
3. 给任务加超时/最大重试，避免死任务永占租约。
4. worker 启动方式定下来（lifespan 内 `asyncio.create_task` 单进程，或 screen 独立进程），并在 `/api/health` 暴露“worker 最近心跳”，让卡死能被发现。

## P1 高（强烈建议接模块前修）

### P1-1 文件上传无大小限制 + 整文件读进内存 → OOM 崩溃

**现象**：上传端点 `content = await file.read()` 一次性把整个文件读进内存，service 里也是 `file_obj.read()` 全量读，**全链路没有任何大小上限校验**。

**证据**：
- [file_transfer.py:22](backend/app/routers/file_transfer.py:22) `content = await file.read()`，只判空不判大小。
- [file_upload_service.py:36](backend/app/services/file_upload_service.py:36) `file_bytes = file_obj.read()` 再次全量入内存。
- 文档宣称的 413“文件过大”其实只在**预览**路径有（[file_preview_service.py:49](backend/app/services/file_preview_service.py:49)），上传路径没有。

**影响**：单机内存有限。知识库一旦接入，同事上传几百 MB 的 PDF/视频，或几个人同时传大文件，进程内存直接爆，整个后端被 OOM kill——这是纯稳定性问题，正是华哥最在意的点。

**修复建议**：
1. 上传端点先校验 `Content-Length` / 累计读取字节数，超阈值（如 200MB，可配）立即 413 中断。
2. 大文件改流式落盘 + 流式算 md5（边读边 `hashlib.md5().update()`），不要一次性 `read()`。
3. 下载侧同理：`download` 用 `FileResponse` 替代 `StreamingResponse(open(...))`（见 P2-5）。

### P1-2 启动同步无错误隔离，一个坏 manifest 拖垮整个后端启动

**现象**：`lifespan` 启动时调用 `sync_apps_from_manifest`，它对每个 `modules/*/manifest.json` 直接 `json.loads(...)`，**无 try/except**。任一模块 manifest 语法错误 → 抛 `JSONDecodeError` → lifespan 失败 → 整个后端起不来。

**证据**：
- [app_service.py:126](backend/app/services/app_service.py:126) `manifest = json.loads(manifest_path.read_text(...))` 裸调用。
- [main.py:30](backend/app/main.py:30) lifespan 内 `await sync_apps_from_manifest(db)` 无保护。

**影响**：框架卖点就是“热插拔模块、物理隔离”。但现在任何一个模块写错一个逗号，就让全平台（含登录、文件、其他正常模块）全部宕机。模块隔离在启动期被打穿。

**修复建议**：逐个 manifest 用 try/except 包裹，坏的记 `module_load_errors`（`/api/health` 已有该字段透出）并跳过，绝不让单模块错误冒泡到 lifespan。孤儿清理也要在“至少成功读到 manifest 列表”的前提下才做，避免读失败时误删。

### P1-3 `create_all` 与 Alembic 双 schema 源，迁移形同虚设

**现象**：`lifespan` 每次启动都 `Base.metadata.create_all`，同时仓里又有 Alembic 迁移。两套 schema 来源并存。

**证据**：[database.py:26](backend/app/database.py:26) `create_all`；[main.py:26](backend/app/main.py:26) 启动即调；迁移目录 `backend/migrations/versions/`（baseline + add_creatable_formats）。

**影响**：`create_all` 只建“缺失的表”，**不会给已存在的表加列**。于是：①新 DB 上 schema 来自模型，迁移没真正跑过、可重放性没验证；②老 DB 上改了模型却忘了迁移，本地因 `create_all` 看似正常，实则线上字段缺失，等接模块加表时一定踩坑。审计明确要求“迁移健康可重放”，当前不满足。

**修复建议**：生产启动去掉 `create_all`，schema 一律走 `alembic upgrade head`；`create_all` 只保留给测试夹具。确认 baseline 迁移能在空库上 `upgrade head` 干净跑通。

## P2 中

### P2-1 去重 / 复制 / 引用计数三者语义不自洽，会泄漏磁盘文件甚至丢内容

**现象 & 证据**：
- `ref_count` 字段被写入，但**永久删除时根本不读它**：删除是否清磁盘，靠的是实时 `COUNT 同 md5 且 deleted=False 的其他记录`（[recycle_service.py:155](backend/app/services/recycle_service.py:155)）。`ref_count` 是装饰性死字段。
- 复制功能 [file_ops_service.py:64](backend/app/services/file_ops_service.py:64) 把内容另存到 `source/{id}.ext`（**非内容寻址**），却沿用 `md5_hash=source.md5_hash`。于是同一 md5 对应两个不同磁盘文件。之后永久删除原件时，`other_refs` 查到这个副本（同 md5）→ 判定“还有引用”→ **跳过删盘 → 原内容寻址文件成为无主孤儿泄漏**。
- 并发竞态：用户A 永久删除某 md5 的最后一份（即将 unlink 磁盘）的同时，用户B 上传相同内容，B 的去重判断 `existing_file and abs_content_path.exists()` 命中→走去重、共享 `storage_path`、**不写盘**；随后 A 提交并 unlink → **B 的记录指向已删除的磁盘文件，内容丢失**。
- 两条同 md5 记录被并发永久删除：READ COMMITTED 下彼此都查到对方仍 `deleted=False` → 都判“有别的引用”→ 都不删盘 → 磁盘文件孤儿泄漏。

**影响**：长期跑下来磁盘空间被孤儿文件吃掉；极端时序下出现“文件还在列表里，点开内容没了”。对一个要长期承载知识库资料的系统，这是数据资产层面的隐患。

**修复建议**：
1. 删除决策改为读权威 `ref_count`，并用 `UPDATE … SET ref_count = ref_count - 1`（原子自减）后判 0 才删盘；上传去重时原子自增。计数与磁盘只认内容寻址路径这一个事实源。
2. 复制必须复用同一内容寻址文件（共享 `storage_path` + 自增 `ref_count`），不要另存 `source/{id}`。
3. 删盘动作放到 `ref_count` 归零的同一事务边界内，并对内容行加锁，消除并发竞态。

### P2-2 Office 补丁乐观锁可被绕过 + 无行锁 → 并发丢更新

**现象**：`apply_patch` 的版本冲突检测依赖客户端传 `source_version_id`；不传则跳过检查；即便传了，读当前版本与写新版本之间对 `pkg` 无行锁。

**证据**：[json_patch_service.py:57](backend/app/services/office/json_patch_service.py:57) `if src_id and …` —— 条件可选；后续读 current → 建新版本 → 改 `pkg.current_version_id` 全程无 `SELECT … FOR UPDATE`。

**影响**：Excel 编辑器要给 Agent + 同事同时操作。两个补丁基于同一版本并发提交，会互相覆盖，后到的赢，前一个改动静默丢失。

**修复建议**：服务端强制要求 `source_version_id` 且必须等于 `pkg.current_version_id`，否则 409；对 `pkg` 行 `FOR UPDATE` 串行化补丁应用。

### P2-3 文件列表分页 `total` 是当前页条数，不是总数

**现象**：`get_file_list` 返回 `total = len(folder_list)+len(file_list)`，即“这一页有几条”，且文件夹根本没分页（一次全返）。

**证据**：[file_service.py:52](backend/app/services/file_service.py:52)。

**影响**：前端无法算总页数，分页 UI 失效；知识库目录上千文件时列表行为错乱。

**修复建议**：用 `func.count()` 单独查真实总数；文件夹与文件统一分页或在契约里说明分页只针对文件。

### P2-4 任意 viewer 可发布全局公告（授权松）

**现象**：`POST /api/notifications/module` 仅需 `viewer`，内部直接调 `create_announcement`，status 写死 `published`，对所有用户可见。

**证据**：[notifications.py:47](backend/app/routers/notifications.py:47) 仅 `require_permission("viewer")`；[notification_service.py:83](backend/app/services/notification_service.py:83) `status="published"`。

**影响**：任何登录用户都能给全公司广播公告，无模块身份校验。内部场景危害有限，但属于授权面不一致。

**修复建议**：该端点要么提权到 editor/admin，要么校验调用方确为已注册模块且声明了通知权限；模块通知与管理员公告应区分来源标识。

### P2-5 下载实现有句柄泄漏与内存峰值风险

**现象 & 证据**：
- 单文件下载 `StreamingResponse(content=open(safe_path,"rb"))`（[file_transfer.py:43](backend/app/routers/file_transfer.py:43)）：对二进制按行迭代、且文件句柄不保证关闭，高并发下载有 fd 泄漏隐患。
- 多文件下载 [file_transfer.py:48](backend/app/routers/file_transfer.py:48) 在 `io.BytesIO` 里把整个 ZIP 建在内存。

**影响**：大文件/多文件批量下载时内存尖峰、句柄堆积。

**修复建议**：单文件用 `FileResponse(path)`；ZIP 用临时文件或流式打包，别全压内存。

## P3 低 / 观察项

- **P3-1 会话语义**：login 每次自增 `session_version`（[auth.py:23](backend/app/routers/auth.py:23)）→ 同一账号换设备登录会把旧设备踢下线（单设备会话）；logout 不动 `session_version`（[auth.py:47](backend/app/routers/auth.py:47)）→ 退出后旧 token 仍有效到 24h 过期。同浏览器多标签共享 token 不受影响。内部场景可接受，确认是否符合预期即可。
- **P3-2 计数用 `len(.all())`**：`search_files`、`get_received_shares` 等用 `len((await db.execute(q)).scalars().all())` 取总数（[file_service.py:173](backend/app/services/file_service.py:173)、[file_share_service.py:121](backend/app/services/file_share_service.py:121)），把全部行拉进内存计数。数据量上来后浪费内存，改 `func.count()`。
- **P3-3 静默吞磁盘异常**：回收站 unlink 失败 `except Exception: pass` 不记日志（[recycle_service.py:169](backend/app/services/recycle_service.py:169) 等3处），磁盘删不掉会无声泄漏，至少加 warning 日志。
- **P3-4 V1 死代码未清**：`backend/scripts/worker/`、`backend/scripts/maintenance/v1_*` 引用已删除的 `app.models.knowledge` 等，无法 import，违反 AGENTS.md 第11条“禁止提交死代码”。因 `scripts/` 不在 AGENTS.md 声明的扫描边界内而长期腐烂。建议删除或迁正。
- **P3-5 NullPool 无连接复用**：每请求新建 asyncpg 连接（[database.py:9](backend/app/database.py:9)）。30 人 + 多条 SSE 长连接时，连接建立开销和 PG `max_connections`（默认100）压力需留意；建议压测确认，或换成小容量 `AsyncAdaptedQueuePool` 并设上限。
- **P3-6 网关 provider 未注入 api_key**：`OpenAIProvider` 在 [router.py:104](backend/app/gateway/router.py:104) 实例化时没传 `api_key`，依赖 url 内嵌鉴权或本地代理。确认 DeepSeek/mimo 的密钥确实通过 url/代理生效，否则云端 provider 会 401。
- **P3-7 未读数算法**：`get_unread_count = 已发布总数 - 用户已读总数`（[notification_service.py:34](backend/app/services/notification_service.py:34)）。公告被 revoke 后已读记录不清理，会让未读数偏小。边角问题。

## 接模块前必修清单

| 优先级 | 事项 | 一句话 |
|--------|------|--------|
| **P0** | 任务队列消费者 | 给 `SystemTaskQueue` 接上真 worker（复用租约模型 + `FOR UPDATE SKIP LOCKED`），否则知识库异步任务全部永久 pending。 |
| **P1** | 上传大小限制 + 流式 | 加上限 + 边读边算 md5、流式落盘，杜绝大文件 OOM。 |
| **P1** | 启动错误隔离 | 单个坏 manifest 不能拖垮整个后端，try/except 跳过并记 `module_load_errors`。 |
| **P1** | 迁移单一事实源 | 生产去掉 `create_all`，统一走 `alembic upgrade head`，验证 baseline 可空库重放。 |
| **P2** | 去重/复制/引用计数收口 | 删除读权威 `ref_count` 原子增减、复制复用同一物理文件、删盘进事务+行锁。 |
| **P2** | Office 补丁强制乐观锁 + 行锁 | 防 Agent 与人并发编辑丢更新。 |

剩余 P2/P3 可在模块接入并行推进期分批清理，不阻塞接入，但建议同窗口排期。
