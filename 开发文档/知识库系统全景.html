知识库系统全景

本文档面向新会话接手与开发排障。读完即可不翻代码理解:实体怎么产生、管线怎么跑、搜索怎么工作、治理怎么闭环。数据取自 owner_id=4 实测(PostgreSQL 库 华世王镞_v2)。

一、系统概览与数据规模

知识库把文档(PDF/图片/表格等)吃进来,经过一条 11 阶段事件驱动管线 逐层加工:解析分块 → 采集原文/OCR/视觉 → 融合正文 → 抽画像 → 抽实体图谱 → 建词项索引 → 建关系。最终产出供 混合检索(关键词 + 向量 + 结构化信号)使用的数据。

两套并行、互不通气的"实体"体系(最容易踩的坑)

任务里常点名的 entity_service.py 不写 kb_terms,它写 kb_entity_dictionary。kb_entity_aliases/kb_disambiguation 都是从体系 A 派生的。

只有一个 worker 任务类型 kb_pipeline_stage。每个任务只跑一个阶段,跑完由 dispatcher 的 settlement 回调把被解锁的下游阶段再入队 —— 是事件驱动 DAG,不是线性流水线。DAG 顺序在 pipeline_service.py:532 settle_pipeline_stage_successors() 用命令式 if 分支定义。lane 决定并发限流:local_preprocess/derived_index/relation_build 内存吃紧时降到 0 并发;llm_analysis/vision_analysis 视为云调用不受本机内存限制。

五、治理子系统

治理分两条线,别混:"实体/证据"治理线 和 "运行时/生命周期"债务线,互不相关。共四个服务。

1. governance_service.py — 实体候选人工审核台(核心)

处理 kb_governance_candidates 的人工审批闭环,做实体的确认/驳回/合并/校准,不做算法分类。

函数行号做什么
list_governance_candidates11分页列候选,按 audit_status 筛选
approve_candidate54通过 → 候选标 approved + 实体词典 status=confirmed
reject_candidate85驳回 → 候选标 rejected + 实体词典 status=archived
merge_entities116合并:源名转别名、迁移关联和证据、写 kb_entity_merge_log
calibrate_extraction243校准:改 entity_name 或 category,同步词典 —— 分类人工纠错的唯一入口

2. derived_governance_service.py — 派生侧表回填(纯算法)

从已有产物回填检索侧表,入口 backfill_derived_governance:267,支持 dry_run/预览/计数,幂等。

  • 别名 _backfill_entity_governance:540:遍历实体词典,_alias_candidates:61 从括号内容、/｜,;、 分隔符、CJK 去空格变体抽别名,过滤低信号词,写 kb_entity_aliases。
  • 消歧:建倒排表,同一词形归属 ≥2 实体时判歧义,写 kb_disambiguation。
  • 另有产物血缘 _backfill_artifact_lineage:357、结论证据链 _backfill_conclusion_evidence:416。
刻意留的口子(347 行注释写死):不自动写 kb_catalogs(分类目录)和 kb_entity_merge_log —— "分类目录需算法出处/反馈字段才能自动写,应先派生成候选而非直接改主分类"。这正是给自动分类功能留的接线位。

3. lifecycle_debt_service.py — 源文件生命周期债务

管源文件丢失的文档:audit_lifecycle_debt:119 扫描源文件已删/丢失/路径非法,archive_source_unavailable_documents:150 带确认令牌 ARCHIVE_SOURCE_UNAVAILABLE 归档。跟实体分类无关。

4. pipeline_debt_service.py — 管线任务队列债务

管任务队列历史垃圾:_classify_task:70 按错误家族(doc_missing/parser_no_content/greenlet_spawn 等)分类,classify_pipeline_lifecycle_debt:668 分类,apply_pipeline_lifecycle_debt_action:977 带确认执行归档/重试/释放。这里的 category 是任务债务类别,不是实体类别。

要做"存量实体批量重分类"往哪加: 扩展 derived_governance_service.py 新增 _backfill_entity_category 回填函数,挂到 backfill_derived_governance 开关体系下,产出到 kb_governance_candidates(category=推断值/confidence=模型分),走 governance_service.approve_candidate 人工确认后再落主词典。不要新建服务、不要直接写主词典字段 —— 存储字段、候选表、审批闭环、回填骨架、router 能力全都齐了。

七、现存问题

问题 1 · term_type 全是 "term",分类逻辑从未实现。 cognitive_index 是纯正则管线,extract_terms 只分词不分类;唯一写入点 cognitive_index_service.py:506 从不给 upsert_term 传 term_type,而它默认就是 "term"。term_type 只是预留的死维度(有唯一约束 uq_kb_terms_owner_norm_type 和索引,架构上本该当分类用)。要分类必须在 derive_document_cognitive_index 里加类型判断再传给 upsert_term。
问题 2 · status 的 rejected 过滤对 kb_terms 形同虚设。 kb_terms.status INSERT 硬编码 'active'(:383),全 backend 没有任何代码把它改成 rejected。搜索侧三处过滤 status='active' 恒为真。词项无淘汰机制 —— 治理只对体系 A 生效。(注:实测 owner 数据里有 2089 条 rejected,来自后续批量治理脚本手动改写,非管线产生。)
问题 3 · rejected 实体在重排序打分里没被过滤(精度漏洞)。 verify_and_score_results 的 entity_stats CTE(1875-1886)完全没 JOIN kb_entity_dictionary、无 status 过滤。93,134 条指向 rejected 实体的 chunk_entities 仍在污染 7704 个文档的 graph_score / term_entity_score,把虚高分推上去挤掉真相关文档。P0 修复:该 CTE 加 JOIN kb_entity_dictionary ... AND ed.status IN('candidate','confirmed')。
问题 4 · 存量分类堆积待审,confirmed=0。 kb_entity_dictionary 10.3 万条全是 candidate,confirmed 为 0;kb_governance_candidates 24.2 万条已带 LLM 类别(confidence 全 0.70)全部 pending。当前缺的不是再跑 LLM,而是审核+入库+去噪。分层治理建议:先规则去噪 ~20 万,再消化 24 万存量候选,再 LLM 只精分去噪后高频 9265 个(186 批),低频 42 万不投 LLM 靠向量兜底。
问题 5 · 别名 3031 条只用于召回,没做正向查询扩展;文档说明与 MCP 工具双缺口。 用户查主名时不会自动带别名去检索,别名增益基本浪费。知识库没有一份主开发文档级别的说明文档(README 仅 88 行、开头英文、漏 24 个能力、表无字段说明);MCP 工具 8 个全服务于"管线跑没跑完",没有一个服务于"跑出来的东西对不对"(实体质量/搜索质量/图谱质量)。
