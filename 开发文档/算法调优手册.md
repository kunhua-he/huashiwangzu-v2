# 算法调优手册（engine算法升级指南）

> 给后续 agent：Agent engine里的算法都会随实际使用需要调整。**要升级某个算法前，先来这里找它的条目**，按 `诉求 → 参数在哪 → 怎么查数据诊断 → 怎么调 → 怎么验证` 五步走。改完按对应批次的验收铁律（真重启 + 真调用，不只 pytest）复核。
>
> engine代码在 `modules/agent/backend/engine/`；记忆/经验存储在 `modules/memory/`；模型配置在 `backend/data/config/models.json`。所有算法都遵守稳定铁律：**崩了只掉精度不崩对话**（见 `投递箱/engine架构总纲`）。

## 通用诊断入口

- **对话级诊断**：每次 `engine.装配上下文()` 返回诊断信息（用了多少 token、各段占比、是否触发压缩、召回/经验命中数），写进后端日志。**调任何算法前先看这个**，确认瓶颈在哪一段。
- **事件日志**：`agent_events` 表（append-only）是对话的唯一事实源，可重放复盘"当时上下文长什么样"。
- **常用库**：`memory_records`（事实记忆）、`memory_links`（记忆链）、`memory_experiences`（成功经验）。

## 全局升级方案：最短探索路径与异步学习

### 目标

这套升级解决的是 Agent 主链路“重复探索、上下文过重、工具结果未被使用、学习阻塞回复”的系统性浪费。优化目标不是限制工具调用次数，而是让每次工具调用更接近答案，让 Agent 复用历史成功路径，只探索当前未知部分。

原则：

- **不硬编码业务内容**：不得把品牌、模块、问题答案写死在路由或 prompt 里。
- **不以工具次数为硬限制**：工具可多用，但每一步必须服务于当前 stop condition。
- **先找历史最短落点，再扩展探索**：相似问题先命中 path recipe，不重新发现路线。
- **SQL/脚本承担确定性探索**：复杂查询和工程化落点交给可复用 SQL、视图、Python recipe。
- **自动学习永远异步**：主回复完成后再蒸馏路径、记忆、偏好，不能阻塞当前轮回答。
- **崩了只降精度不崩对话**：路径匹配、recipe、后台学习失败时回退到现有 Agent 流程。

### 目标架构

```text
用户当前问题
-> 轻量意图/签名提取
-> 查询历史 path recipe
-> 命中：执行最短可验证路径
-> 未命中：进入受诊断约束的常规探索
-> 当前证据满足 stop condition 后立即回答
-> trace 入库
-> 后台 distiller 异步沉淀新 path recipe / 经验 / 记忆
```

关键点：

- 主链路只负责完成当前请求。
- 学习链路只消费 trace，不参与同步回答。
- 路径复用失败不是错误，只是回到原流程。
- 新路径必须在成功完成后沉淀，不能把失败或半成品路径提升为 recipe。

### Path Recipe

Path recipe 记录的是“怎么到达答案”，不是“答案是什么”。它应该描述某类问题的最短可验证步骤、所需参数、停止条件和历史质量。

建议数据形态：

```json
{
  "intent": "kb_product_list",
  "signature": "询问某主体/品牌有什么产品或产品线",
  "required_inputs": ["entity_name"],
  "best_path": [
    "knowledge.entity_lookup",
    "knowledge.product_index_query",
    "compact_answer"
  ],
  "stop_condition": "已拿到产品系列、具体产品名、来源摘要",
  "fallback_path": ["knowledge.search"],
  "avg_latency_ms": 900,
  "success_rate": 0.96,
  "tool_result_usage_rate": 0.9,
  "active": true
}
```

适合优先沉淀的 recipe：

| intent | 典型问题 | 最短落点 |
|---|---|---|
| `capability_status` | “你有生图能力吗” | capability registry cache |
| `tool_task_status` | “这个大概要多久/好了没” | recent tool result -> task/record status |
| `file_resolution` | “图片的绝对路径/file_id 是什么” | file detail / publish state |
| `kb_product_list` | “某品牌有什么产品” | entity/product index -> top source chunks |
| `module_contract_probe` | “这个工具能不能被 Agent 调用” | manifest + live capability registry |
| `coding_known_fix` | “类似上次的问题再修一下” | previous successful recipe -> changed files -> focused verification |

### 建议表与 SQL

新表应属于 Agent/Memory 体系，命名可在实现时按模块边界决定。下面是逻辑模型，不要求一次性全部建表。

| 表 | 用途 |
|---|---|
| `agent_trace_runs` | 每轮请求的总体 trace：intent、耗时、token、是否命中 recipe、最终状态 |
| `agent_trace_steps` | 每个探索步骤：工具、SQL、脚本、耗时、结果摘要、是否被答案使用 |
| `agent_path_recipes` | 可复用路径 recipe：签名、步骤、所需输入、停止条件、质量指标 |
| `agent_path_outcomes` | recipe 每次执行结果：成功/失败、失败原因、是否回退 |
| `agent_tool_result_cache` | 最近工具结果缓存：task_id、record_id、file_id、capability schema |
| `agent_learning_queue` | 异步学习候选：trace_id、候选类型、优先级、去重 key |

典型查询：

```sql
-- 找某类 intent 最快且稳定的路径
SELECT id, intent, best_path, stop_condition, success_rate, avg_latency_ms
FROM agent_path_recipes
WHERE intent = :intent
  AND active = true
ORDER BY success_rate DESC, avg_latency_ms ASC
LIMIT 3;
```

```sql
-- 找经常探索但没被答案使用的步骤
SELECT step_name, COUNT(*) AS unused_count
FROM agent_trace_steps
WHERE result_used_in_answer = false
GROUP BY step_name
ORDER BY unused_count DESC
LIMIT 20;
```

```sql
-- 判断某条 recipe 是否该降权
SELECT recipe_id,
       COUNT(*) AS runs,
       AVG(CASE WHEN success THEN 1 ELSE 0 END) AS success_rate,
       AVG(latency_ms) AS avg_latency_ms
FROM agent_path_outcomes
WHERE created_at > now() - interval '14 days'
GROUP BY recipe_id
HAVING COUNT(*) >= 5
ORDER BY success_rate ASC, avg_latency_ms DESC;
```

如果使用向量签名匹配，recipe 表应有 `signature_embedding`，查询按相似度和质量联合排序：

```sql
SELECT id, intent, best_path, stop_condition
FROM agent_path_recipes
WHERE active = true
ORDER BY
  signature_embedding <-> :query_embedding ASC,
  success_rate DESC,
  avg_latency_ms ASC
LIMIT 1;
```

### Python Recipe

复杂探索应沉淀为确定性脚本，而不是让模型每轮临场拼 SQL、猜表名、猜接口。

建议目录：

```text
dev_toolkit/recipes/
  kb_product_lookup.py
  image_task_status.py
  file_resolution.py
  capability_lookup.py
  module_contract_probe.py
  trace_waste_report.py
```

设计要求：

- 输入参数显式，输出 JSON。
- 只读脚本默认安全；写操作必须单独命名并要求更高权限。
- 输出包含 `evidence`、`confidence`、`stop_condition_met`。
- 脚本失败返回结构化错误，Agent 可回退常规流程。
- 脚本结果能被 trace 标记是否进入最终回答。

示例输出：

```json
{
  "success": true,
  "intent": "kb_product_list",
  "evidence": [
    {"source": "kb_chunks", "id": 123, "summary": "产品系列与具体产品"}
  ],
  "answer_facts": {
    "series": ["祛斑美白系列", "补水修护系列"],
    "products": ["娇韵皓芙霜", "素滢精华液"]
  },
  "confidence": 0.91,
  "stop_condition_met": true
}
```

### Prompt 协议

全局 Agent prompt 应加入探索协议，而不是提示“少用工具”。建议规则：

```text
1. 先提取当前问题的 intent、required_inputs、stop_condition。
2. 查询是否命中历史 path recipe；命中则优先执行 recipe。
3. recipe 缺参数、失败或 stop_condition 未满足时，才扩展探索。
4. 每个探索步骤前说明它要验证哪个未知点。
5. 已满足 stop_condition 时立即回答，不继续做锦上添花的探索。
6. 工具结果没有进入答案或决策，trace 中标记为 unused。
7. 自动学习、路径蒸馏、经验写入只能进入后台队列，不能阻塞回复。
8. 后台学习失败不影响当前对话。
```

这套协议应全局生效，模块只提供能力和 recipe，不在模块里硬编码对话策略。

### 异步自动学习

自动学习主链路禁止同步执行。正确链路：

```text
当前轮完成
-> trace run/steps 入库
-> learning queue 产生候选
-> 后台 worker 去重、评分、蒸馏
-> 写入 path recipe / memory_experiences / user preference
-> 更新 recipe 质量指标
```

同步允许的只有极轻量动作：

- 写 trace。
- 写 learning queue 候选。
- 读取已有 recipe。

异步学习触发条件：

- 成功完成且 stop condition 明确。
- 用户纠正了 Agent 的路径或答案。
- 同类问题重复出现，且存在稳定最短路径。
- 某个 recipe 多次失败，需要降权或停用。
- 工具结果长期 unused，需要生成浪费报告。

不得触发学习的情况：

- 单纯任务状态问答。
- 无效参数校验失败。
- 工具服务不可用导致的临时失败。
- 已有 recipe 成功且没有新增信息。
- 用户没有表达长期偏好的一次性指令。

### 指标

不要用工具调用次数作为硬指标。应跟踪以下指标：

| 指标 | 含义 | 优化方向 |
|---|---|---|
| `path_reuse_rate` | 命中历史 recipe 的比例 | 越高越好 |
| `answer_ready_distance` | 从问题到满足 stop condition 的步骤数 | 越低越好 |
| `rediscovery_rate` | 已有 recipe 但仍走常规探索的比例 | 越低越好 |
| `unused_step_ratio` | 工具/检索结果未被答案或决策使用的比例 | 越低越好 |
| `context_waste_tokens` | 未进入最终决策的上下文 token | 越低越好 |
| `latency_by_intent` | 各 intent 平均/分位耗时 | 关注 P95 |
| `learning_block_ms` | 自动学习阻塞主链路耗时 | 必须接近 0 |
| `recipe_success_rate` | recipe 执行成功率 | 低于阈值降权 |
| `fallback_rate` | recipe 回退常规流程比例 | 过高说明路径不稳 |

浪费审计规则：

- 工具结果没有被最终答案引用或用于决策，标记 unused。
- 同轮重复读取同一 schema/capability，标记 duplicate_discovery。
- 已有 task_id/file_id/record_id 仍重新搜索，标记 rediscovery。
- 自动学习候选被同步蒸馏，标记 learning_blocking。
- 检索 topK 远大于最终使用来源，标记 retrieval_overfetch。

### 落地阶段

P0：Trace 先行，不改行为。

- 为每轮记录 run/step 级 trace。
- 标记 tool result 是否被答案使用。
- 记录 intent、stop condition、recipe 命中情况。
- 输出 `trace_waste_report.py`，先量化浪费在哪里。

P1：Path Recipe 读路径。

- 建 `agent_path_recipes` 与 `agent_path_outcomes`。
- 先人工/半自动沉淀 10-20 条高频 recipe。
- 主链路只读取 recipe，不自动写 recipe。
- 命中失败必须回退原流程。

P2：高频 Python Recipe。

- 优先做 capability、task status、file resolution、kb product lookup。
- 每个脚本必须有 JSON 契约和可复跑验证。
- Agent prompt 改为“先查 recipe，再执行最短落点”。

P3：异步 Path Distiller。

- 后台消费 trace，生成 recipe 候选。
- 去重、评分、成功率门槛通过后才激活。
- 对失败 recipe 自动降权、停用或追加 fail_notes。

P4：全局优化闭环。

- 定期生成浪费审计。
- 用 SQL 找 P95 慢 intent 和 unused 高发步骤。
- 把稳定优化沉淀为 recipe 或脚本。
- 删除不再有效的 recipe，防止路径库膨胀。

### 验收标准

实现这套升级时，每阶段都必须有可验证指标：

- `learning_block_ms` 接近 0，自动学习不阻塞用户回复。
- 高频 intent 的 P95 延迟下降，且答案质量不下降。
- `unused_step_ratio` 和 `rediscovery_rate` 下降。
- recipe 命中失败可回退，不能造成对话失败。
- 新增 recipe 不包含业务硬编码答案，只包含路径、参数、停止条件。
- 所有确定性脚本有可复跑命令和 JSON 输出样例。

参考验收用例：

| 用例 | 预期路径 |
|---|---|
| “某品牌有什么产品” | 命中 `kb_product_list`，走结构化索引/recipe，不宽泛重检索 |
| “你有生图能力吗” | 命中 `capability_status`，读 registry/cache，不装配大上下文 |
| “这个大概要多久” | 命中 `tool_task_status`，读最近 task/record，不重新探索 |
| “图片的绝对路径” | 命中 `file_resolution`，查 file 状态或 publish 状态 |
| “继续上次同类修复” | 命中 coding recipe，只探索变更相关文件 |

## 1. 动态 Token 预算装配（budget_allocator.py）

- **诉求**：本地模型无 KV 缓存，上下文超 ~8万 token 就卡。要把交给模型的上下文控制在预算内，同时尽量塞最相关的内容。
- **参数**：`models.json` 每个模型档的 `context_budget`（整数 token，`null`=不限）。各段优先级与子预算在 `budget_allocator.py`。
- **诊断**：看装配诊断——总 token、各段（系统/近期/召回记忆/知识库/经验/摘要）占比。哪段挤掉了重要内容？哪段占比过大？
- **调优**：本地档调小 `context_budget`（建议 3-5万）；调整各段子预算占比（如召回记忆给太多挤掉近期对话，就降召回 top_k）；必留段永远不动。
- **验证**：设 `context_budget` 后跑长对话，诊断里总 token ≤ 预算、必留段在、关键信息没丢（问开头的事能答上）。

## 2. layered_memory·语义召回（modules/memory recall）

- **诉求**：换个说法也能召回相关记忆（第10轮召回第3轮），零硬编码分类。
- **参数**：召回相似度阈值、`top_k`、是否 rerank、ILIKE 兜底——都在 `modules/memory/backend/router.py` 的 `_recall`/`_match` 附近常量。
- **诊断**：SQL 看 `memory_records`（`embedding` 是否非空、`summary` 质量、`confidence`/`recency_score`）。召回结果带 `similarity`（0-1 余弦）和 `rerank_score`——命中相关吗？相似度分布合理吗？
- **调优**：召回不全→降相似度阈值/调大 top_k；召回噪音多→提阈值/强制 rerank；摘要质量差→换更好的便宜模型蒸馏。
- **验证**：存一条记忆，用**完全不同措辞**召回能命中（关键词召不回的向量能）。

## 3. 即时融合（memory.fuse）

- **诉求**：一个查询召回多条原始记忆时，临时用便宜模型融成贴题简报，比硬拼接信号高、还省 token。
- **参数**：触发条件（召回条数 > 阈值且预算紧）、用哪个便宜模型——`modules/memory` fuse 相关 + `models.json` 摘要档。
- **诊断**：诊断信息看"召回记忆"段是走了融合还是直接拼总结层；对比融合前后 token 与相关度。
- **调优**：融合太频繁费钱→提触发阈值（多数情况直接用总结层）；简报不贴题→改融合 prompt。
- **验证**：构造多条相关记忆的查询，融合输出比拼接更短更贴题；融合失败时降级到总结层拼接不报错。

## 4. 记忆链图（memory_links）

- **诉求**：把跨越多轮对话的同一条线索串起来（语义判定），召回种子能顺链带出相关条。
- **参数**：建链相似度阈值、顺链跳数、链权重——`modules/memory` dream 建链逻辑 + recall 顺链扩展。
- **诊断**：SQL 看 `memory_links`（边数、`weight` 分布、是否把不相关的也连了）。
- **调优**：链太密（噪音）→提建链阈值；该串的没串→降阈值/增跳数（注意别爆预算）。
- **验证**：存几条同主题分散记忆，dream 跑后 `memory_links` 有边，召回种子顺链带出相关条。

## 5. dream 自优化（modules/memory 后台任务）

- **诉求**：长期保持记忆库/经验库干净，不变成一团乱麻——合并重复、建链、衰减旧记忆、剪枝低质经验。
- **参数**：合并相似度阈值（当前 >0.92）、建链阈值（>0.55）、衰减周期（30 天）、触发频率（每 N 轮/定时）——`modules/memory` dream 常量。
- **诊断**：跑前后对比记忆/经验总数、重复率、`confidence`/`recency_score` 分布、停用经验数。
- **调优**：重复没合干净→降合并阈值（注意别把不同的也合了）；旧记忆该忘没忘→缩短衰减周期；dream 太重→降频率/分批。
- **验证**：故意存近重复记忆，dream 跑后合并/降权、库不膨胀；dream 崩了不影响主对话（fire-and-forget）。

## 6. experience_memory加权（memory_experiences）

- **诉求**：把摸索过的成功路径存下来，下次类似请求语义命中直接走；成功加权、失败降权+注释，越用越准。
- **参数**：去重相似度阈值（`EXPERIENCE_DEDUP_THRESHOLD`）、注入相似度阈值、净值算法（`success_weight - 失败惩罚`）、成功判定信号、失败停用阈值——`modules/memory` experience 相关 + `engine.py` 成功判定。
- **诊断**：SQL 看 `memory_experiences`（`success_weight`/`fail_count`/`fail_notes`/`active`）。哪些经验常被注入但常失败（该降权/停用）？哪些高价值（该优先）？
- **调优**：注入了没用的经验→提注入阈值/调净值公式重罚失败；成功判定不准（把失败当成功加权）→改 `engine.py` 成功信号（当前是"工具无 error + 正常到 [DONE]"，可加便宜模型自评）；好经验被去重误并→提去重阈值。
- **验证**：存经验→换措辞命中→照走成功则 `success_weight`+1、失败则 `fail_count`+1 且 `fail_notes` 有内容、净值低被剪枝。

## 7. compressor（compressor.py，批4）

- **诉求**：上下文超预算时，保头尾摘中间，信息不丢只是粒度变粗。
- **参数**：头部保留 N、尾部保留 M、折叠比例（渐进 0/10/20/50/100%）、摘要字符上限、便宜模型——`compressor.py` + `models.json`。
- **诊断**：SQL 看 `agent_events` 的 `compaction` 事件（被折叠区间 + 摘要质量）；问对话开头的事是否还答得上（信息丢没丢）。
- **调优**：摘要丢了关键信息→加大头/尾保留、改摘要 prompt；还是超预算→提高折叠比例；工具对被拆散报错→确认成对处理。
- **验证**：长对话触发压缩，`compaction` 事件落库、原始事件仍在、token 在预算内、开头信息没丢。

## 8. 模型fallback_chain（fallback_chain.py / gateway，批4）

- **诉求**：主模型挂了不停服，自动降到备用/本地，最多精度降。
- **参数**：`models.json` 的 `fallback_chain`（顺序）、触发降级的失败类型（超时/5xx/403/空响应）——`backend/app/gateway/` 或 `fallback_chain.py`。
- **诊断**：日志搜"降级:xxx→yyy"，看降级频率与原因；确认 `fallback_chain` 各档真实可调。
- **调优**：主模型常失败→换更稳的主档或调整链顺序；本地兜底慢→链尾放更快的本地档。
- **验证**：模拟主模型失败（改错 key/断网），对话仍出结果、日志有降级记录。

## 9. stuck_detector（stuck_detector.py，批4）

- **诉求**：模型抽风重复同一工具/同一报错时打断，不无限烧 token。
- **参数**：连续重复阈值 N（默认3）、窗口大小——`stuck_detector.py` 常量。
- **诊断**：日志看是否有粘滞打断记录；有没有误杀（正常的重复操作被当粘滞）。
- **调优**：误杀→提高 N/放宽指纹；放过了死循环→降 N。
- **验证**：构造重复同一工具，N 次后被打断、给提示、不无限循环。
