# R6 算法研究专线：图片、视频、知识库算法路线与产品目标

日期：2026-07-03  
Agent：codex-r6-algorithm-research  
范围：只读研究为主；未修改主项目代码。证据来自 AGENTS.md、开发文档入口、framework/backend/module 文档、`modules/knowledge/README.md`、`modules/media-intelligence/README.md`、`modules/image-vision/`、`modules/media-asr/`、`modules/knowledge/backend/services/*`、活栈 health/probe、`/Users/hekunhua/Documents/Agent/reference_sources` 本地参考目录，以及少量官方开源 README。

## 0. 一句话结论

不要把 R6 做成“堆模型/堆 VLM”的任务。华世王镞 V2 当前真正要达成的是：让企业文件、图片、视频都能变成可追溯、可搜索、可引用、可治理、可评测的证据网络。算法自研重点应该放在系统编排、分层证据、成本门控、去重与失效治理、评测反馈闭环；OCR、ASR、解码、镜头切分、目标检测、VLM、向量索引等底层模型/引擎优先接成熟库或框架网关。

## 1. 产品/系统目标：为什么要做这些能力

业务目标不是“识别图片/识别视频/做知识库”本身，而是让 Agent 和用户能对企业资料做可信问答、追溯证据、复用素材、减少重复劳动。

目标可以拆成四层：

1. **资料可入库**：PDF、Office、图片、视频、音频都能登记为统一知识资产，处理状态可见，失败可解释。
2. **内容可找回**：用户问问题时，系统能从文字、图片文字、视频字幕、视觉线索中召回正确片段。
3. **答案可追溯**：结果必须带来源文件、页码、时间戳、截图/帧、OCR box、chunk/page fusion/entity evidence，而不是只给一段模型总结。
4. **成本可控且可评测**：本地确定性算法先跑，成熟小模型补充，VLM/LLM 只用于高价值语义判断；每阶段有准确率、延迟、成本、失败率指标。

## 2. 当前项目现状对照

### 2.1 knowledge：已有真实基础，但检索和多媒体还没到稳态

已有基础：

- 文档登记、解析、分块、向量、原始采集、页级融合、实体图谱、治理候选已形成主链路。
- `knowledge` 能力已注册：`search/get_block/get_page_fusion/get_entity_dictionary/get_graph_context/get_pending_count/get_ocr_words/ingest/get_ingest_status/export` 等。
- 表结构已有 `kb_documents`、`kb_chunks`、`kb_raw_data`、`kb_page_fusions`、`kb_pipeline_runs`、实体/图谱/治理表。
- `search_service.py` 当前实现是关键词 ILIKE + 向量余弦 + RRF，可选 rerank。
- `raw_collection_service.py` 有三轮采集：文本、OCR、视觉构成；tesseract 可用时能出词级坐标，否则回退 VLM OCR。
- `fusion_service.py` 有页级融合与启发式 fallback，并把融合正文重建为 chunks。
- `pipeline_orchestrator.py` 有 stage 注册、stale 检测、source unavailable、stage run 诊断，方向正确。

主要问题：

- 向量检索目前把所有有 embedding 的 chunk 拉到 Python 算 cosine，小规模可用，但数据一大就是性能瓶颈。稳态应使用 pgvector/LanceDB 类索引做 server-side ANN/metadata filter。
- 关键词检索是 ILIKE + 简单词频，不是真 BM25/FTS；中文分词、短词、同义词、字段权重还弱。
- chunking 有 `title_aware/structure_aware` 服务，但主路径里 `chunk_and_embed` 仍以固定切分为核心，后续要统一到 DocumentIr 结构感知 chunk。
- LLM/VLM 调用集中在 raw/fusion/entity/profile，已有日志诊断，但成本预算、缓存、失败降级策略还需要产品化指标。
- 视频只是 README 规划，尚未真正把 video segment / timestamp evidence 接入 `kb_chunks` 与检索引用。

### 2.2 image-vision：低成本本地图片分析已经相对扎实

已有基础：

- `image_analysis.py` 用 Pillow 做尺寸、格式、模式、sha256、方向、动画、透明度、亮度、对比度、饱和度、边缘密度、主色、近似空白、aHash/dHash、EXIF orientation。
- `should_use_vlm` 已经按 local/semantic/auto 做 VLM 门控：小图标、低细节图、透明图、本地可判断时不走 VLM；photo_like/detailed_graphic 或用户语义模式才走 VLM。
- `router.py` 通过 `run_uploaded_file_capability` 读框架文件，符合 `check_file_access` 安全边界；还能把结果尝试写入 `content:store_analysis_resource`。

短板：

- 没有真实 OCR、目标检测、版面识别、质量评分中的 blur/noise/compression 维度。
- 感知哈希只有 aHash/dHash，可增加 pHash、裁剪/缩放鲁棒去重、近重复聚类阈值。
- 该模块和 `media-intelligence` 功能有重叠，后续应避免两套图片事实层并行分叉。

### 2.3 media-intelligence：契约骨架成立，但大量 placeholder/degraded

已有基础：

- 模块边界、provider registry、统一结果 schema、HTTP endpoints、capabilities、sandbox 都建立了。
- pipeline 分层清楚：`local_algorithms -> small_model -> vlm_refine`。
- 文件访问通过 `run_uploaded_file_capability`，边界正确。
- local layer 对图片有基本 metadata，对视频有 ffprobe metadata 和 timeline keyframe markers。

必须明确：

- README 也写明“deterministic placeholders”。
- `local_algorithms.py` 的 OCR 返回 `ocr_engine_missing`，对象检测返回 `object_detector_missing`。
- `small_model.py` 是 rule-based degraded summary。
- `vlm.py` 是 `vlm_refine.not_configured`。
- `embed_image` 当前是 average_hash 类 local fingerprint，不是语义 embedding。

因此它不是 fake-success 的最佳例子，因为状态和 warnings 是 degraded；但如果上层只看 HTTP `success:true`，就会被误读为“分析完成”。后续验收必须检查 `stages[].status`、`degraded`、`warnings` 和产物非空。

### 2.4 media-asr：已有真实 ffprobe/ffmpeg/mlx_whisper 链路，但未接入 knowledge

已有基础：

- `probe_media_file` 用 ffprobe 校验音轨/视频流/时长，限制 4 小时，避免坏输入进入昂贵模型。
- `extract_audio_from_video` 用 ffmpeg 抽音轨，支持采样率和格式。
- `transcribe_audio_file` 用 `mlx_whisper`，输出 `text` 和 timestamp segments。
- `build_segment_blocks` 已能把 ASR segments 变为 blocks。
- 能力 `extract_audio/transcribe_audio/transcribe_video` 已注册，editor 角色，因可创建框架文件。

短板：

- 当前是独立工具模块，不是 knowledge video ingest 的一环。
- Whisper 模型适合 Apple Silicon 本地快速落地，但中文企业视频长期可考虑 FunASR/SenseVoice/WhisperX 等成熟库作为可替换 provider。
- 分段结果还缺持久化到 video segment 表、chunk 时间戳、字幕/OCR/关键帧 evidence 的统一结构。

## 3. 图片分析算法路线

### 3.1 本地优先能做什么

第一层应尽量本地、确定性、可缓存：

- 文件事实：mime、扩展名、大小、sha256/md5、EXIF、orientation、dpi、宽高、像素数、色彩模式、动画帧数。
- 视觉统计：主色、亮度、对比度、饱和度、透明像素比例、边缘密度、空白/低信息判断。
- 质量评分：blur（Laplacian variance/OpenCV）、噪声、过曝/欠曝、压缩伪影、分辨率是否低于阈值、是否截图/图标/照片。
- 去重：aHash/dHash/pHash，多尺寸缩放鲁棒近重复；同 md5 直接复用，同 pHash Hamming distance 做疑似重复聚类。
- OCR：优先 PaddleOCR/Tesseract；输出全文、行/词、box、置信度、语言；截图/海报/表格图片必须进入 OCR 文本检索。
- 版面：图片中的表格、文档截图、PPT 页面、二维码、图表区域；第一阶段只需 region type + box，不要急着做复杂图表理解。
- 目标检测：成熟库或模型输出 label/box/confidence；企业内要先定义对象词表，不要泛化识别一切。

### 3.2 什么时候才需要 VLM

VLM 只在以下条件触发：

- 用户显式要求语义理解、描述场景、解释图表、判断关系/用途。
- 本地 OCR/检测/质量分不能解释主体，例如照片、复杂设计稿、产品场景图。
- 图片是高价值资料，且本地事实显示有足够信息量：非空白、非小图标、非低分辨率装饰图。
- 多源冲突需要裁决：OCR 读到文字但版面/主体不清，或图表需要语义概括。
- Agent 答案需要“看图证据”，必须返回 resource_id、局部 box/region、VLM prompt 和模型版本。

### 3.3 自研 vs 接库

应自研：

- 图片分析结果 schema、VLM 门控策略、成本预算、缓存、resource 存储、去重阈值和企业语料评测。
- OCR/检测/哈希/质量/VLM 之间的融合逻辑，以及“可用于知识库检索的文本/证据”转换。

优先接成熟库：

- OCR：PaddleOCR/Tesseract。
- 质量/哈希：Pillow/OpenCV/imagehash。
- 检测/分割：YOLO/RT-DETR/SAM2 等，但 Ultralytics AGPL 需许可证审查或商业授权。
- VLM：框架网关或本地 Qwen/MiniCPM 系列，不在项目内训练基础模型。

## 4. 视频分析算法路线

### 4.1 第一性目标

视频入库的第一版不是“看懂整部视频”，而是：上传视频后，用户能搜索到视频中的一句话、一页 PPT、一段字幕、一个关键画面，并跳转到时间点。

### 4.2 推荐 pipeline

1. ffprobe：读取 duration、fps、codec、分辨率、音轨、字幕轨、bitrate，拒绝坏输入。
2. segment：第一版固定 30 秒窗口；同时保留 scene cut/ASR sentence 时间戳，后续合并为 topic segment。
3. audio：ffmpeg 抽音频，ASR 输出 timestamped segments；这是视频 RAG 的主干信息源。
4. keyframes：ffmpeg/decord/PyAV 抽真实帧，不要只返回时间 marker。第一版每段 first/mid/last 或 scene representative。
5. OCR/subtitle：读取内嵌字幕；关键帧跑 OCR，去重连续相同字幕/PPT 页面。
6. objects/scenes：第二阶段接检测/场景分类，只做可检索标签和证据，不做复杂视频理解前置。
7. VLM segment caption：只对高价值片段、OCR/ASR 为空但视觉信息丰富、用户追问片段时触发。
8. persistence：segment_text、visible_text、caption、keyframe_ref、time_start/end、confidence、cost、dependency_status 全部持久化。
9. retrieval：检索结果必须返回 `file_id/document_id/segment_id/start/end/source_type`，支持跳转。

### 4.3 成本控制

- 默认 ASR + OCR + 关键帧，不默认 VLM 全视频扫。
- 建立 content_hash/frame_hash/transcript_hash，重复视频或重复片段跳过。
- 分段级缓存，失败可重试单段，不重跑全视频。
- 每个视频设置 max_duration、max_frames、max_vlm_segments、max_cost。
- VLM prompt 要带本地事实，只补语义，不重复 OCR/ASR。

### 4.4 推荐接库

- ffprobe/ffmpeg：继续作为第一层事实与音频抽取底座。
- PyAV：容器/流/时间戳精细控制，适合作 fallback 与复杂视频。
- decord：批量随机抽帧，适合 segment keyframes。
- PySceneDetect：第一版可选 scene cut，解释性强；TransNetV2 第二阶段再考虑。
- ASR：当前 `mlx_whisper` 可作为 Apple 本地 MVP；中文企业稳态优先评估 FunASR/SenseVoice，英文或对齐/说话人可评估 WhisperX/faster-whisper。
- OCR：PaddleOCR 优先。
- VLM：Qwen/MiniCPM 等本地/网关模型只做 gated caption/refine。

## 5. 知识库算法路线

### 5.1 解析与 IR

继续坚持 DocumentIr / Content IR：parser 产出结构块，knowledge 只消费 IR。不要让 knowledge 直接实现 PDF/DOCX/XLSX parser，也不要让视频绕开 IR 思路。视频可以扩展为 MediaIr 或 DocumentIr 的 timeline blocks：`segment`, `transcript`, `frame`, `ocr_region`, `caption`。

### 5.2 Chunking

短期：统一走 structure/title aware chunk，保留 fixed_size 兼容。  
稳态：按文档类型选择策略：

- 文档：标题层级 + 段落 + 表格完整性。
- 表格：行列/区域块，不把表格硬切碎。
- 图片：OCR/描述/metadata 分开成块。
- 视频：ASR sentence/topic segment + OCR text + VLM caption 融合为 segment chunk。

验收重点不是 chunk 数，而是召回是否命中正确证据。

### 5.3 Embedding、Hybrid、Rerank

当前 BGE-M3 维度与 pgvector 基础存在，但检索仍需升级：

- 向量检索应从 Python 全量 cosine 迁到 pgvector/LanceDB 类索引。
- 关键词应从 ILIKE 升级到 PostgreSQL FTS/pg_trgm/BM25 方案，中文需分词或 ngram 兜底。
- Hybrid 用 RRF 保留，但要区分 text/chunk/page/entity/video segment 的来源权重。
- rerank 不应默认全开，只对 top 20-50 候选、复杂查询、低置信召回开启。
- 检索结果必须带 stage provenance：keyword/vector/page_fusion/entity/media_asr/media_ocr/vlm_caption。

### 5.4 页级融合与实体图谱

页级融合是当前知识库最有产品辨识度的部分：不是只把 parser 文本扔进向量库，而是把文本、OCR、视觉构成做交叉印证。后续不要推翻，应增强：

- 明确每轮证据的权重和失败状态。
- fusion 结果必须保留 evidence ids 和冲突摘要。
- 实体抽取从融合层做是对的，但要引入实体规范化、别名、同义词、跨文档合并候选。
- GraphRAG 不应成为 MVP 前置；先让实体证据链和跨文件关系稳定。

### 5.5 增量更新、去重、失效源

已有 stale/source_unavailable/pipeline_runs 基础，后续补强：

- 源文件 hash 变更只重跑受影响 stage。
- 删除/回收站文件必须让检索、看板、图谱都排除历史 done 结果。
- 相同 md5 直接复用解析/ASR/OCR；近重复图片/视频片段走感知 hash 和 transcript hash。
- pipeline 所有 stage 都要有 `done/degraded/failed/skipped` 可解释口径，调用方禁止只看 HTTP success。

### 5.6 评测集与在线反馈

必须建立最小评测集，否则算法路线会变成凭感觉：

- 文档 QA：20-50 个真实问题，标注正确 chunk/page。
- 图片：20 张覆盖图标、截图、照片、表格、海报、低质图、重复图。
- 视频：3-5 个 1-5 分钟样本，覆盖会议讲话、PPT 录屏、带字幕视频、无字幕画面。
- 指标：Recall@5、MRR、答案引用准确率、OCR 字符准确率、ASR WER/CER、重复识别 precision/recall、每文件处理成本、P95 延迟、degraded 可解释率。
- 在线反馈：用户点“有用/无用/引用错/漏召回”，落到 query-result-feedback，周期性生成 hard negatives 和 rerank/权重调参集。

## 6. 目标树

| 业务目标 | 系统能力 | 算法组件 | 验收指标 |
|---|---|---|---|
| 用户能问企业资料并得到可追溯答案 | 混合检索 + 证据引用 | parsing、chunking、embedding、BM25/FTS、RRF、rerank | 文档 QA Recall@5 >= 0.8；引用页/块准确率 >= 0.8 |
| 图片资料可被检索和去重 | 图片本地事实层 + OCR + VLM 门控 | EXIF/尺寸/主色/质量/hash/OCR/对象/semantic caption | 本地分析 P95 < 1s；低价值图 VLM 调用率 < 10%；重复图召回 >= 0.9 |
| 视频可入库、可搜索、可跳转 | 视频 segment evidence | ffprobe、ffmpeg、ASR、OCR、keyframes、segment chunks | 3 个样例视频 100% 生成 timestamp chunks；搜索命中正确时间段 Recall@5 >= 0.75 |
| 成本不爆 | 分层执行 + 缓存 + 限额 | stage cache、hash dedupe、VLM gating、budget | 每文件 VLM 调用数有上限；重复文件重跑节省 >= 80% |
| 知识可治理 | 实体词典、图谱、候选审核 | entity extraction、alias merge、relation scoring | 每个实体有 evidence；候选可审核；跨文件关系有相似度与原因 |
| 生产可维护 | pipeline 诊断与失效处理 | stage run、source unavailable、stale propagation | 任一失败 stage 可在 ingest-status 看到原因；无假 done |

## 7. 阶段计划

### MVP：1-2 天

目标：打通“最便宜但可用”的多媒体知识入口，不追求复杂理解。

任务：

1. 图片：把 `image-vision` 的本地分析作为图片事实层基准；media-intelligence 图片能力若继续保留，必须复用或对齐 `image-vision`，不再扩展另一套 placeholder 图片算法。
2. 视频：用 `media-asr:transcribe_video` 输出 timestamp segments；先把 segment text 作为 knowledge 可检索 chunk 的设计/接口方案定稿。
3. media-intelligence：明确 degraded placeholder 不作为生产完成信号；验收改看 `stages[].status` 和 artifacts 非空。
4. 知识库：建立最小评测集：10 个文档问题、10 张图片、3 个短视频；记录 expected source。

可测指标：

- 图片本地分析 20 张样例全部返回 dimensions/hash/quality/color，且 auto 模式低价值图不触发 VLM。
- 视频 3 个样例中，ffprobe 能读取 metadata，ASR 输出 timestamp segments；坏输入返回结构化 4xx。
- knowledge search 对 10 个文档问题 Recall@5 >= 0.7，结果不包含 source unavailable 文件。
- 所有新增验收只打活栈，不造长期测试数据；上传样例用完清理或放模块 test-data。

### 稳态：1 周

目标：从“能用”到“可信、可扩展、可诊断”。

任务：

1. 图片：接入 PaddleOCR/Tesseract provider；补 pHash、blur/noise/compression quality；把 OCR 文本进入 knowledge chunks。
2. 视频：真实抽帧落资源；固定 segment + 可选 PySceneDetect；关键帧 OCR；ASR/OCR/字幕合并去重。
3. 检索：pgvector server-side topK；关键词升级到 FTS/pg_trgm/BM25 方案；RRF 权重可配置；rerank gated。
4. pipeline：每个 media stage 持久化状态和 artifact hash，支持单段重试和 source unavailable。
5. 评测：扩到 50 个 QA、50 张图片、10 个视频片段，形成自动评测脚本/工具台能力。

可测指标：

- 图片 OCR 字符级准确率在人工标注小集 >= 0.85；pHash 重复识别 precision >= 0.9。
- 视频 segment 搜索 Recall@5 >= 0.75；每分钟视频默认 VLM 调用数为 0。
- 1,000 chunks 检索 P95 < 500ms（不含 rerank）；10,000 chunks 有明确指标基线。
- pipeline degraded/failed 都可在 ingest-status 和 stage diagnostics 看到原因。

### 增强：2-4 周

目标：做出多模态知识网络的产品差异。

任务：

1. VLM gated segment caption：仅对低文本、高视觉价值片段生成 caption；支持用户追问时按需补跑。
2. 视觉 embedding：接 open_clip/Visualized-BGE 或网关向量；用于图片/关键帧相似检索。
3. GraphRAG 增强：实体别名、跨文档关系、视频片段实体、证据链问答；借鉴 microsoft-graphrag 但不照搬全套前置。
4. 在线反馈闭环：把错召回/漏召回/错引用变成 hard negatives 与权重调参数据。
5. 成本面板：按 module/stage/provider/file 展示耗时、调用、失败、缓存命中。

可测指标：

- 多模态查询（“某个画面/某句发言/某页PPT”）Recall@5 >= 0.8。
- VLM caption 只覆盖 <= 20% 高价值 segment，但带来 >= 10% 的视频检索召回提升。
- 用户反馈样本每周能产出 rerank/权重调整报告。
- 重复媒体二次入库成本降低 >= 80%。

## 8. R6/R7/R8 分工建议

### R6：算法路线与 MVP 接缝

- R6-A 图片事实层收束：image-vision 与 media-intelligence 图片能力对齐，定 schema 和 VLM 门控指标。
- R6-B 视频 segment IR：定义 video segment blocks / metadata / evidence / timestamp 引用，不急着大改 UI。
- R6-C 检索评测集：最小 QA/image/video gold set + Recall@K/MRR 评测脚本或工具台能力。
- R6-D placeholder 治理：media-intelligence degraded 语义进入验收，避免 HTTP success 被当成算法完成。

### R7：生产级适配器与索引

- 接 PaddleOCR、真实 keyframe extraction、PySceneDetect 可选、ASR provider 抽象。
- knowledge 检索升级：server-side vector index + FTS/BM25 + rerank gate。
- 把 video ASR/OCR segments 真正写入 knowledge chunks，支持时间点跳转。

### R8：增强理解与闭环优化

- VLM gated caption、视觉 embedding、跨模态检索。
- GraphRAG/实体图谱增强，但基于已有 evidence/pipeline，而不是推倒重建。
- 在线反馈、成本面板、自动调权/重排数据闭环。

## 10. 风险清单

1. **成本黑洞**：knowledge raw/fusion/entity 现在会调用 VLM/LLM；视频如果逐帧 VLM 会迅速失控。必须默认 local/ASR/OCR，VLM gated。
2. **假完成风险**：media-intelligence 返回 HTTP success 但 stage degraded；上层必须读 stage 状态。
3. **性能瓶颈**：Python 全量向量 cosine 不能支撑大库。
4. **重复建设**：image-vision 和 media-intelligence 图片事实层重叠，需收束。
5. **许可证风险**：Ultralytics AGPL 不应未经审查直接进企业产品。
6. **多 worker 状态**：媒体长任务所有状态、锁、进度必须落库或原子文件，不可用内存。
7. **评测缺口**：没有 gold set 时，算法升级无法证明提升。

## 11. 最终建议

优先顺序：

1. 先把图片/视频都变成 knowledge 可检索证据，而不是追求“看懂一切”。
2. 先复用成熟库和当前框架网关，少自研底层模型。
3. 自研系统层：schema、pipeline、gating、cache、dedupe、evaluation、feedback。
4. 用指标驱动阶段推进：Recall、引用准确率、处理成本、degraded 率、P95 延迟。
5. R6/R7/R8 拆线程时按“接缝设计 → 生产适配器 → 增强理解”推进，别让一个线程同时背图片、视频、检索、图谱全部改动。
