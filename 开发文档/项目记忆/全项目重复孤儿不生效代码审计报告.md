# 全项目重复逻辑 / 孤儿代码 / 不生效代码审计报告

> 审计日期: 2026-06-30
> 审计范围: backend/app, backend/tests, frontend/src, modules, 开发文档
> 审计方法: codegraph 调用链 + 路由扫描 + 能力扫描 + DB schema + SQL 只读查询 + probe 活栈验证

## 1. 总结结论

1. **最大重复业务线：文件内容流转存在 3 套并行体系** — Old JSON Package (`framework_file_json_*`, 0 数据)、Content Package (`framework_content_packages`, 55 条)、Artifact (`framework_artifacts`, 12+ 条)。Office route 已声明"桥接到 Content Package"但实际 `/api/office/status/{id}` 返回 500。

2. **三个导出/发布入口行为不一致** — `content:export` → `content:publish`(创建 artifact)、`office-gen:generate_to_artifact`(生成+创建 artifact+解析 content package)、`artifact:export/publish`(已 artifact 的发布)。同名冲突策略、版本记录、content package 绑定状态各不相同。

3. **`framework_file_json_packages` + `framework_file_json_versions` + `framework_file_json_patches` + `framework_file_json_tasks` 四张表全部 0 记录** — 旧 JSON Package 体系已事实废弃，但 ORM 模型 (`backend/app/models/office.py:7-80`) 和旧 Office service 残存文件仍在。

4. **`/api/office/status/{file_id}` 返回 HTTP 500** — 桥接到 Content Package 的代码存在运行时错误（Probe 验证：`GET /api/office/status/1` → 500 Internal Server Error），用户/Agent 调用该端点会直接崩溃。

5. **`/api/office/patch/preview`、`/api/office/patch/apply`、`/api/office/rollback` 三个端点返回 `{"result":"deprecated"}`** — 仍在路由注册但已空壳，前端/runtime 仍有调用代码指向这些端点。

6. **7 个 parser 模块 + knowledge 自身 parse = 至少 8 套 parse 实现** — 同一格式（如 xlsx 由 `xlsx-parser` 和 `knowledge/parsing_service` 两套路径解析），无 canonical 指定。

7. **每个模块 runtime/index.ts 重复 150+ 行相同代码** — `apiGet`/`apiPost`/`authHeaders`/`_handle401`/`getApiUrl` 在 15+ 模块中完全重复，这是结构性问题。

8. **`excel_versions` 表 0 记录** — excel-engine 版本管理能力注册了但从未产生版本数据。

9. **Agent 工具指引中 `content:*` 系列不是独立 module capability（没有 content module），而是框架 REST 路由** — 工具发现可能无法通过 capability 列表正确列举 content 能力。

10. **当前 Git 工作区有 43 个修改文件（+2547/-657 行）** — 包含 `office_export.py` 删除、旧 office 服务删除、大量模块重写，审计结果部分可能已在该分支修复，但未提交。

## 2. P0 问题

### P0-1: `/api/office/status/{file_id}` 返回 500

- **位置**: `backend/app/routers/office.py:60-78`
- **证据**: Probe `GET /api/office/status/1` → HTTP 500 "Internal server error"。codegraph 显示 `get_status` 调用 `_require_file_access` → `check_file_access`，该函数返回 `{"accessible": bool, "permission": str}` 但第 44 行读 `access.get("accessible")`，而 `file_service.py:164-177` 的 `check_file_access` 返回的是 `File` ORM 对象而非 dict，类型签名不匹配。
- **复现方式**: `curl -X GET /api/office/status/1`（任意 file_id）
- **影响**: 所有通过 `/api/office/status` 查文档状态的调用（含 Agent 工具调用）都会崩溃。
- **建议修复**: 修复 `_require_file_access` 中 `check_file_access` 返回值处理；或删除该桥接路由直接指向 content package API。

### P0-2: `framework_file_json_packages` 旧体系 ORM 模型仍可被调用但数据为零

- **位置**: `backend/app/models/office.py:7-80` (FileJsonPackage, FileJsonVersion, FileJsonPatch, FileJsonTask)
- **证据**: SQL `SELECT count(*) FROM framework_file_json_packages` → 0; `framework_file_json_versions` → 0; `framework_file_json_patches` → 0; `framework_file_json_tasks` → 0。模型在 `backend/app/models/__init__.py` 中导入导致 `Base.metadata` 包含这些表定义。
- **复现方式**: SQL 查询表 count；检查 `models/__init__.py` 的 import。
- **影响**: 如果代码路径误写入旧表，数据会丢失；如果旧 service 仍被 import 但调用失败，造成假绿。
- **建议修复**: 删除 `backend/app/models/office.py` 中的旧模型（或标记 deprecated），从 `models/__init__.py` 移除 import，DROP 四张空表。

### P0-3: knowledge 与 content pipeline 两套独立 parse 路径

- **位置**:
  - `knowledge:ingest` 能力 → `modules/knowledge/backend/services/document_service.py` → `parsing_service.py`（内部用 VLM/docling 解析）
  - `content:pipeline` 路由 → `backend/app/services/content/pipeline_service.py` → 调用 parser 模块 (`docx-parser`, `xlsx-parser` 等)
- **证据**: codegraph 显示两套 parse 调用链完全不同。knowledge:ingest 自己读盘+解析+分块，content:pipeline 走 parser 模块能力调用。Probe 验证 `POST /api/content/pipeline` (file_id=1) 成功，`knowledge:ingest` (file_id=99999) 返回 skipped。
- **复现方式**: 对比 `document_service.py` 和 `pipeline_service.py` 的 parse 调用链。
- **影响**: 同一文件经两个入口得到不同 block 结构，知识库检索的 chunk 与 content package 的 block 不一致。
- **建议修复**: 将 knowledge ingest 的 parse 阶段桥接到 content:pipeline（或 parser 模块），消除两套解析输出。

### P0-4: `check_file_access` 函数签名不一致

- **位置**: 
  - `backend/app/services/file_service.py:164-177` → 返回 `File` ORM 对象，不存在时 raise `NotFound`
  - `backend/app/services/file_share_service.py:9-28` → 返回 `{"accessible": bool, "permission": str}` 字典
- **证据**: `office.py:41-46` 的 `_require_file_access` 调用 `check_file_access` 后尝试读 `access.get("accessible")`，但 `file_service.py` 版本返回的是 File ORM 对象，运行时抛 `AttributeError`。
- **复现方式**: 调用任何通过 `file_service.check_file_access` 然后按 dict 方式取 `.get()` 的代码路径。
- **影响**: 权限校验路径产生 500 而非正确的 403/404，安全审计假设 fail-closed 被 bypass。
- **建议修复**: 统一两个函数的返回签名，或给 `file_service.check_file_access` 加明确返回类型。

## 3. P1 问题

### P1-1: 三个导出入口行为不一致

| 入口 | 创建 Artifact | 创建 Content Package | 冲突策略 | 版本记录 |
|------|---|---|---|---|
| `content:export` → `content:publish` | ✅ create_artifact | ✅ (通过 pipeline) | create_version | ✅ artifact_versions |
| `office-gen:generate_to_artifact` | ✅ create_artifact | ✅ (显式调用 pipeline) | create_version | ✅ artifact_versions |
| `artifact:export/publish` | ✅ 自身 | ❌ 不创建 | auto_rename | ✅ artifact_versions |
| `excel-engine:export_xlsx/publish_to_desktop` | ❌ 直接物理文件 | ❌ 不创建 | 未知 | ❌ excel_versions 0 记录 |

- **位置**: `backend/app/services/content/export_service.py:38-154`, `modules/office-gen/backend/router.py:407-526`, `backend/app/services/artifact_service.py:641-750`
- **风险**: Agent 靠随机选导出入口，行为不可预测。

### P1-2: 7 个 parser 模块(8 套实现)无 canonical

- parser 模块清单: `docx-parser`, `pptx-parser`, `xlsx-parser`, `pdf-parser`, `text-parser`, `csv-parser`, `structured-parser`, `email-parser`, `markdown-parser`
- 每套都做：读盘 → 解析 → 输出 `{file_id, format, blocks, resources}`
- knowledge `parsing_service.py` 做同样的工作但输出到 `kb_chunks`/`kb_raw_data`
- `content:pipeline` 调用 parser 模块输出到 `framework_content_packages`
- **证据**: codegraph 显示 parser 模块与 knowledge parsing_service 调用链完全独立。
- **风险**: 9 份 parse 代码维护，格式支持不一致，block schema 漂移。

### P1-3: Agent 工具指引中 `content:*` 非 module capability

- `content:pipeline`, `content:export`, `content:publish` 等是 framework 路由, 不在 `/api/modules/capabilities` 列表中
- Agent 的工具发现 (`tool_discovery.py`) 通过 `list_capabilities()` 枚举模块能力, 无法发现 content 能力
- `capabilities()` 输出中无 content 模块
- **证据**: 对比 `routes(filter=content)` 输出和 `capabilities()` 输出
- **风险**: Agent 需要硬编码提示词才能知道 content API 存在

### P1-4: `excel_versions` 表 0 记录

- SQL: `SELECT count(*) FROM excel_versions` → 0
- 所有 excel-engine 版本管理能力 (`list_versions`, `restore_version`) 都无法产生实际数据
- `excel_history` 有数据但 `excel_versions` 始终为空

### P1-5: runtime/index.ts 150+ 行重复 boilerplate

- 15+ 模块各有完全相同的 `apiGet`, `apiPost`, `apiPut`, `authHeaders`, `_handle401`, `getApiUrl`, `getCurrentUser` 等函数
- 每个文件 150-250 行，总计 ~3000 行重复代码
- 存在多种变体(`modules.call` 的 `parameters` vs `params` 字段名在不同模块不同)

### P1-6: `office-gen:generate_to_artifact` 验证产生测试数据未清理

- Probe 调用 `office-gen:generate_to_artifact({"format":"xlsx","filename":"test_audit"})` 产生了 artifact_id=13, file_id=668, content_package_id=56
- 当前无自动清理机制

## 10. 不建议现在动的内容

1. **Artifact 体系 (`framework_artifacts` + `artifact_service`)** — 虽然与 Content Package 部分重叠，但 Artifact 专注"发布产物版本管理"，是 Content Package 的下游消费者。保留。

2. **Parser 模块** — 虽然 7 个 parser 各自独立，但 content:pipeline 依赖它们作为实际解析引擎。指定 content:pipeline 为 canonical 即可，不需要合并 parser。

3. **`excel-engine` 状态式内存模型** — 与 `xlsx-parser`/`office-gen:xlsx` 不重复，excel-engine 做实时编辑状态管理，后两者做导入/导出。但版本管理路径(`excel_versions` 0 记录)需要修。

4. **当前工作分支的 43 个已修改文件** — 审计发现的部分问题（如 old office service 删除、office_export 删除）已在当前分支修复。不要跨分支改动，等待该分支合并后再清理。

5. **`framework_private_modules` 和 `framework_app_registry`** — 模块注册与发现的基础表，虽然有数据但归属框架组件，不视为重复。
