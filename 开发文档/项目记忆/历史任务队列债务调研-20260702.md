## 当前代码判断
- 新版 `modules/knowledge/backend/services/source_file_state.py` 已能把 DB 文件行不存在/已删除分为 `source_file_missing/source_file_deleted`。
- 新版 `modules/knowledge/backend/services/pipeline_service.py` 在 handler 入口和异常分支都会检查 source availability，不可用时调用 `mark_document_source_unavailable` 并返回 `status: skipped`。
- `backend/app/services/task_worker.py` 只把 `status in {failed,error}`、`success=false` 或裸 `error` 当失败；`status=skipped` 会被记为 completed。因此大量旧 `File not found` 任务如果通过安全 retry 机制重跑，理论上会落为 completed/skipped 而不是继续 failed。

## 建议给修复会话
1. 不要直接删除 failed 行；先加一个只对历史任务生效的治理工具/脚本，支持 dry-run。
2. 对 `kb_pipeline + File not found` 且 `file_row missing/deleted` 的历史 failed：可批量重置为 pending 或实现专门 archive 标记，让新版 handler 统一写 `parse_error=source_file_missing/source_file_deleted` 并完成为 skipped。
3. 对 `doc_deleted` 对应任务：建议归档为 historical/obsolete，不必重试。
4. 对 `file_row_live` 的 13 条：物理文件存在，应抽样重试或直接检查当前 parser 是否可解析；不要归为源文件缺失。
5. 对 `Parser returned no content blocks`：保留为真实解析质量债，优先抽 5 个文档看扩展名/内容，决定是 parser 空结果应 skip、还是解析器 bug。
6. 对 `Document is already parsing`：这是并发/状态锁旧债，建议历史归档；同时确认当前 `_find_inflight_pipeline_task` + parse 状态恢复足够。
7. 对 `profile_evolve No module named init_db`：旧代码债，当前代码已修；可按 conversation 去重重跑一次，或归档。如果重跑，建议去重同 conversation，仅保留每个 `(conversation_id, owner_id)` 最新一条。
8. 对 `Failed to parse profile JSON`：建议改 handler 为“LLM 输出不可解析时记录 skipped/soft failure，不占用 failed 队列”，或增强 JSON 修复/二次解析。
9. 对 `__emergency08_missing_handler__` 和 `conversation_id=999999`：明确标记为测试债，可归档。
