# 调研结论
本次只做深度调研，不投递下一封信。当前审计修复08尚无收件箱回信，活系统 `/api/system/status` 仍报旧 worker false，而 `/api/health` 报 worker running true，因此必须先等08完成并验收。

# 关键证据
- git dirty_count 约 39，横跨 backend/app、dev_toolkit/smoke、modules/agent、modules/excel-engine、modules/image-vision、文档与新增 content IR 文件。
- `/api/health`：worker running true，task_queue pending 1、failed 730。
- `/api/system/status`：worker.status false，message 仍为 Background worker process not found。
- `framework_system_task_queues`：failed 大头为 `kb_pipeline/File not found` 568+、历史 `profile_evolve/No module named 'init_db'` 130。
- `framework_event_log`：file.uploaded 大多 completed，历史 failed 仅 2 条旧 schema 错；主要问题是事件派生任务失败而非 event_bus 本体。

# sandbox 证据
- 发现 19 个模块有 sandbox，只有 8 个有 `sandbox/test_module.py`。
- 直接 `python3 modules/*/sandbox/test_module.py` 时 excel-engine 因缺 sqlalchemy 失败、pdf-parser 因缺 pdfplumber 失败。
- 使用 backend venv 后 pdf-parser 通过；excel-engine 需要 `PYTHONPATH=backend backend/.venv/bin/python modules/excel-engine/sandbox/test_module.py` 才通过。
- 说明 README/验收入口不可复现，适合下一封信治理“模块 sandbox 验收矩阵”。

# 其他风险
- 文件访问：多数 parser 已走 `check_file_access`，但 `backend/app/services/office/text_editor_service.py` 与 `csv_editor_service.py` 服务层内部仍裸 `db.get(File)` 后读盘，目前靠 router 外层权限检查兜底，服务复用时边界脆弱。
- TypeScript：`any/as any` 主要集中在 viewer 全局 payload、wechat-writer、excel-engine、image-gen catch；前端 `npm run build` 通过，但这仍是契约债。
- content IR：dirty 面大但已有 `backend/tests/test_content_ir_architecture.py` 覆盖，优先级低于队列失败分类、sandbox 验收入口、文件权限内聚。
