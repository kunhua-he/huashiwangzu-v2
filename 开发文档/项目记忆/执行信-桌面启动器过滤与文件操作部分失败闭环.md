## 一句话目标

把桌面工作台里“用户点了没反应/批量操作明明部分失败却提示成功”的体验问题收口：启动器不再把不可打开后台模块当普通应用，文件批量操作必须显示成功/失败明细。

本任务只做桌面交互和文件操作反馈，不做后端资产清理、不改 release gate、不改 Knowledge/Agent 后端。

## 目标 1：启动器不再误导用户打开后台模块

现状问题：

```text
后端注册 enabled app 很多，其中部分 show_in_launcher=false 或 window_type=background。
启动器/命令注册可能把它们当普通应用展示。
用户点击后 openWindow 返回 null 或无反馈。
```

要求：

1. 启动器普通应用列表只展示可直接打开的 app。
2. 后台能力/不可直接打开模块如果进入搜索结果，必须标注“后台能力/不可直接打开”，并提供说明而不是无反馈。
3. `openWindow` 返回 null 时，调用侧必须显示 toast/提示：

```text
该能力是后台服务，不能直接打开窗口
```

4. 搜索结果要区分：

```text
应用
文件/最近项（如已有）
后台能力
命令/动作（如已有）
```

不要为了完成目标重构整个启动器，只做必要收口。

## 目标 2：批量文件操作部分失败必须可见

现状问题：

```text
paste/copy/move 中单项失败 catch 后跳过，最后可能仍显示成功。
```

要求：

1. 批量 copy/move/paste/delete 这类操作返回结构化结果：

```ts
interface BatchOperationResult {
  successCount: number
  failCount: number
  errors: Array<{ id?: number | string; name?: string; message: string }>
}
```

2. UI 提示必须区分：

```text
全部成功
部分成功：完成 N 个，失败 M 个
全部失败
```

3. 失败项要能展开/查看，至少 console/debug 或 toast 详情里可见。
4. 不允许单项失败被静默吞掉。
5. 保持原有 happy path 行为。

## 目标 3：不和 LoadState 任务打架

本任务不要改 `frontend/src/shared/api/` 的统一 API 错误基础设施。若需要错误格式，只在本任务文件内做局部类型或调用现有 helper。
