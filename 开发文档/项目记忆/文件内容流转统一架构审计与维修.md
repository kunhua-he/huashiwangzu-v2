## 做了什么

审计《文件内容流转统一架构第一次探索报告》的实现状态，并就地维修发现的问题。

### 审计结论

**报告核心内容已基本实现：**
- ✅ `framework_content_packages` + `framework_content_package_versions` 表已创建
- ✅ `framework_resources` + `framework_resource_refs` 表已创建
- ✅ `backend/app/models/content.py` — ContentPackage/ContentPackageVersion/Resource/ResourceRef 模型
- ✅ `backend/app/schemas/content_package.py` — ContentPackageIR/ContentBlockSchema/ResourceRefSchema 标准 IR
- ✅ `backend/app/services/content/` — PackageService/PipelineService/ExportService/ResourceService/Adapter 全套
- ✅ 14 个 `content:*` 能力已注册并可通过 `/api/modules/call` 调用
- ✅ 16 个 REST 端点 (`/api/content/packages/...`) 可用
- ✅ `file.uploaded` 事件自动触发 content pipeline
- ✅ `kb_documents` 已有关联 `content_package_id` 的逻辑

**依然存在的问题：**
- ⚠️ Framework Office Services (`backend/app/services/office/`) 仍然是平行实现，未合并到 parser 模块
- ⚠️ Parser 模块不实际提取内嵌资源到 `framework_resources`（只返回 metadata）
- ⚠️ Agent 尚未暴露 content:* 工具层
- ⚠️ `framework_file_json_packages` 和 `framework_file_json_versions` 仍存在
- ⚠️ `excel_workbooks` 编辑态与 content package 无同步机制

### 就地维修

1. **Bug: kb_documents content_package_id 查询** — `modules/knowledge/backend/services/document_service.py` 中 `not ContentPackage.deleted` 是 Python 布尔值短路，不会生成 SQL `WHERE deleted = false` → 改为 `ContentPackage.deleted.is_(False)`
2. **Bug: export_adapter xlsx** — `pass` 死代码导致 `adapter_payload["sheets"]` 不被赋值 → 删除 `pass`
3. **Bug: export_adapter pptx** — 固定 `adapter_payload["slides"] = []` 永远空 → 新增 `_blocks_to_slides()` 方法
4. **标准化：7 个 parser 模块统一中文类型为英文** — `"标题"→"heading"`, `"段落"→"paragraph"`, `"表格"→"table"`, `"图片"→"image"`, `"代码"→"code"`，对齐 `ContentBlockSchema.BlockType` 枚举
5. **Bug: markdown-parser 缩进** — `flush_list()` 内 `blocks.append(...)` 多缩进一级 → 修正

### 验证

- ruff lint 全部通过（8 个 parser + 1 个 service + 1 个 document_service）
- 后端重启后 health ok
- `/api/content/packages` 返回 200
- 14 个 `content:*` 能力全部在 `/api/modules/capabilities` 列表中
