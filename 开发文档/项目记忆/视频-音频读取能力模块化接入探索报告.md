## 探索结论

### 方案
新建模块 `media-asr`，三个能力不分拆模块：
1. `extract_audio` — 视频提取音频（ffmpeg 子进程）
2. `transcribe_audio` — 音频转文字（mlx-whisper MLX 本地推理）
3. `transcribe_video` — 视频直接转文字（内部串联前两个能力）

### 已确认可复用框架能力
- `check_file_access` (file_service.py:164) — 文件权限校验
- `run_uploaded_file_capability` (uploaded_file_runner.py:15) — 标准 parser 能力封装
- `register_capability` / `call_capability` / `list_capabilities` (module_registry.py) — 能力注册与发现
- `SystemTaskQueue` + `register_task_handler` (task_worker.py:31) — 后台长任务（长音频兜底）
- `create_artifact` (artifact_service.py:130) — 产物保存（转录文本）
- Agent 自动发现：`skill_list/skill_describe/skill_use` 零修改

### 参考的模块模式
- text-parser / image-vision：`run_uploaded_file_capability` + `register_capability` 标准样板
- office-gen converter.py：`asyncio.create_subprocess_exec` + `shutil.which` + `TemporaryDirectory` + 超时 — 完美 ffmpeg 模式
- office-gen router.py：`_save_file_bytes` / `_save_to_artifact` 产物保存模式

### 环境状态（全部就绪）
- Python 3.14.4
- ffmpeg 8.1 (Homebrew)
- mlx-whisper 0.4.3 (已 pip 安装，Apple Silicon + MLX 原生加速)

### 依赖声明
- mlx-whisper 进模块自己的 requirements.txt（不写框架 requirements.txt）
- ffmpeg 在 health 端点动态检测（`shutil.which("ffmpeg")`）

### 风险点
- 长视频耗时：>10min 音频应走 task queue，而非同步超时
- 临时音频文件：用 `TemporaryDirectory` 上下文管理器，确保清理
- 多 worker：model 加载各 worker 各自一次（mlx 在 GPU 上跨进程不共享），无竞态
- 子进程安全：使用 `asyncio.create_subprocess_exec` 避免 shell 注入
- 输出过长：返回不包含原始音频 bytes，只返回文本；`text` 字段按需截断或分页

### 不做抖音链路
账号安全风险、扫码登录 UI 复杂度、接口不稳定、平台封号风险。确认本轮只做本地文件能力。
