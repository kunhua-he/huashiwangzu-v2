# 调研信：全模块能力缺口与重复造轮子总审计

## 任务定位

这是一封只读深度调研信，适合与“全链路产品化落地总攻大信”并发执行。原则上不改代码，只输出审计报告和后续执行建议，避免与正在开发的大信发生 Git 冲突。

## 目标

一次性审计 V2 全模块能力缺口、重复造轮子、跨模块边界风险、V1 参考能力缺口，并形成一份可直接拆给后续执行会话的总清单。

禁止修改 V1，禁止删除文件。

## 必做审计项

1. 全模块 manifest 审计：entry 类型、component_key、public_actions、permissions、是否 background-service 却可打开窗口。
2. 全模块 README 验收矩阵缺口：哪些模块缺矩阵、哪些矩阵是空话、哪些没有 sandbox 状态。
3. 跨模块调用边界：查是否存在模块互相 import 业务代码、直接读写对方表、绕过 capability registry。
4. 文件访问安全：查所有 file_id 读取链路是否使用 `check_file_access`。
5. Content/Knowledge/Agent 重复造轮子：解析状态、引用结构、artifact 元数据、任务状态是否重复定义。
6. 前端类型安全：查 `any`、`as any`、`@ts-ignore`、字段名不匹配、`转中文()` 被用于数据字段。
7. 后端假绿风险：查 `success=false`、error-only dict、semantic failure、任务队列 completed 但业务失败。
8. 测试污染风险：查测试创建文件/Knowledge/Package/Artifact 后是否清理。
9. V1 能力缺口：只读对照 V1，列出 V2 仍缺的用户可见关键能力。
10. 分析模块群重复：PDF/DOCX/PPTX/XLSX/CSV/TXT/Markdown/Image/Media 是否各自造 IR，是否能统一。

## 输出文件

写报告到：

开发文档/项目记忆/全模块能力缺口与重复造轮子总审计报告.md
```

8. 哪些问题适合总攻大信继续做，哪些必须单独收口
```
